## Context

这次问题只发生在 `apps/web` 的历史会话侧栏交互层。当前 `清空历史会话` 的业务前提、禁用条件和后端 API 都是正常的：菜单项可见且启用，程序化 `click()` 能打开确认框，单条删除也能正常弹出确认层并发请求。真正失效的是“真实鼠标点击更多菜单中的批量清空项”这条事件路径，它在 `pointerdown` 之后、`click` 之前就提前失活，最终表现为菜单收起但确认框不出现。

这说明问题不在 store 或后端契约，而在 `SessionRail` 内部对菜单、确认层和展开面板的局部状态管理。当前组件同时处理 `pointerdown` 外部关闭、`focusout` 关闭、hover 展开/收起和危险确认互斥状态，任何一处对瞬时焦点或目标命中的误判，都可能在用户完成点击前把菜单项卸载。

## Goals / Non-Goals

**Goals:**
- 让 `更多操作 -> 清空历史会话` 在真实鼠标点击下稳定打开确认框，而不是在 `click` 前丢失激活。
- 保持批量清空菜单项在可用状态下可被标准键盘焦点导航访问和触发。
- 收敛菜单、确认层和展开面板之间的关闭语义，使局部交互状态在真实浏览器事件顺序下仍然可预测。
- 为这类菜单内危险操作补上能覆盖真实指针激活顺序的组件测试。

**Non-Goals:**
- 不修改 `apps/agent-backend` 的批量清空接口、请求参数或数据删除语义。
- 不重做历史侧栏的整体视觉、hover 展开模型或单条删除确认样式。
- 不引入新的通用弹层库、事件库或第三方测试依赖。
- 不借由这次修复增加新的兜底路径或静默 fallback。

## Decisions

### Decision: 将批量清空菜单项的激活视为一次原子交互，禁止关闭逻辑在 `click` 前抢先回收菜单状态

`SessionRail` 需要把“菜单项被用户实际点击”视为一个完整交互，而不是允许 `pointerdown`、焦点漂移或局部关闭逻辑在中途先把 `activeSurface` 复位。实现上应优先保证菜单项在用户完成主指针激活前仍然保持可命中、可点击，确认层打开后再进入下一阶段状态切换。

Rationale:
- 当前故障已经证明逻辑层和确认层本身是通的，失败点在真实点击未能走完。
- 对危险菜单项来说，先保证激活动作闭环，比在点击中途过早追求“及时关闭菜单”更重要。
- 这能把问题局限在 `SessionRail` 本地状态机，不需要把危险确认搬到全局 modal 管理。

Alternatives considered:
- 继续依赖现有 `pointerdown`/`focusout` 组合，仅微调样式或文案：无法从根本上解决点击在 `click` 前失活的问题。
- 恢复为 `window.confirm`：能绕开竞态，但会回退掉已经建立的产品内确认体验。

### Decision: 用稳定的交互边界判断替代对瞬时焦点状态的脆弱假设

关闭历史侧栏菜单或展开面板时，应基于稳定的交互边界判断当前事件是否仍发生在 `rail` 的有效交互域内，而不是依赖可能短暂失真的 `document.activeElement` 或对过渡性焦点状态的微任务推断。对于菜单、确认层、触发按钮和展开面板之间的切换，判断逻辑必须容忍真实浏览器里 `pointerdown -> focus change -> click` 的顺序。

Rationale:
- 当前回归最像是局部关闭逻辑在焦点切换期间错误认定“用户已离开交互域”。
- 稳定边界判断比瞬时焦点猜测更贴近真实用户意图，也更容易为测试构造确定性断言。

Alternatives considered:
- 继续使用当前基于微任务的焦点收敛方式，只在少数分支上加例外：容易留下新的隐藏竞态。
- 把所有局部表面全部 Teleport 成全局层：复杂度偏高，超出这次修复范围。

### Decision: 回归测试必须覆盖真实菜单点击链路，而不仅是程序化 `element.click()`

组件测试需要显式覆盖以下事件路径：打开 `更多操作` 菜单、对 `清空历史会话` 执行真实指针序列、断言确认框出现；同时保留取消、确认、Esc 和点击外部关闭的断言。仅验证程序化 `click()` 不足以捕获这类菜单内激活回归。

Rationale:
- 这次故障正是单测只覆盖了程序化 `click()`，没有捕获浏览器真实事件序列差异。
- 测试约束住事件顺序后，可以防止后续再因菜单关闭逻辑调整而回归。

Alternatives considered:
- 只补手工验证说明：不可回归，也无法防止后续重构再次引入同类问题。

## Risks / Trade-offs

- [修复菜单项激活竞态时误伤现有的点击外部关闭语义] → 将测试同时覆盖菜单内激活与菜单外关闭，确保两类路径被分别锁定。
- [为了解决批量清空而引入另一套特例，导致单条删除确认逻辑继续分叉] → 以 `SessionRail` 的局部交互状态机为边界，优先抽象共同的命中/关闭判断，避免单独给批量清空写隐式旁路。
- [键盘可达性修复后改变当前焦点返回行为] → 把焦点返回视为显式验收点，确认菜单关闭与确认框关闭后焦点落点仍然可预测。

## Migration Plan

1. 在 `SessionRail` 内梳理菜单、确认层和展开面板的局部状态转换，找出会在 `click` 前回收菜单项的关闭路径。
2. 调整菜单项激活与关闭顺序，确保 `清空历史会话` 的真实鼠标点击能稳定进入确认框状态。
3. 补充回归测试，覆盖真实菜单点击、键盘聚焦与关闭语义。
4. 该变更只影响前端交互，无需后端迁移或数据迁移；若需回滚，只需恢复 `SessionRail` 相关交互实现。

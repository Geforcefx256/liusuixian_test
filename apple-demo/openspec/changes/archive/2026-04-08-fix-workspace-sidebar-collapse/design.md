## Context

当前 `apps/web` 的 workbench shell 已经维护了三类与右侧侧栏可见性相关的状态：用户手动折叠、受限宽度自动折叠，以及在自动折叠期间手动重新展开。问题不在状态缺失，而在状态没有完整映射到布局和视觉层：

- `WorkspaceSidebar` 接收了 `collapsed`，但该状态没有实现为真实的隐藏布局。
- `WorkbenchShell` 会在宽度不足时触发自动折叠，但重新展开后的侧栏仍按桌面固定宽度 chrome 渲染，窄宽度下容易出现顶部控件拥挤和“拼装感”。
- 现有测试覆盖了自动折叠与自动折叠后的重开，但没有覆盖用户主动折叠、折叠后保留活动文件、以及受限宽度下手动重开的完整行为链。

这次变更只涉及前端 workbench shell、workspace sidebar 和相关样式/测试，不涉及后端协议、会话模型、文件 API 或第三方依赖。

## Goals / Non-Goals

**Goals:**

- 让手动折叠成为真实可见的布局切换，而不是仅更新内部布尔状态。
- 保留当前“自动折叠 + 手动重新展开”的交互模型，但明确其优先级和可见性结果。
- 让侧栏在受限宽度下重新展开时仍保持可操作的顶部控件与可理解的布局，不出现明显拥挤或控件漂浮感。
- 通过测试固定手动折叠、重新展开、自动折叠和活动文件保持不变的行为。

**Non-Goals:**

- 不改动 workspace 文件树的数据结构、上传/新建/重命名/删除能力。
- 不扩展“模板”页签为真实模板管理系统。
- 不引入新的布局框架、动画系统或第三方 UI 依赖。
- 不改动 workbench 的会话、编辑器打开/关闭、或 backend workspace contract。

## Decisions

### 1. 由 shell 负责侧栏可见性，折叠时让侧栏真正退出布局

`WorkbenchShell` 已经是 split layout 的拥有者，分隔条、展开按钮和侧栏组件也都由它统一排布。因此折叠结果应由 shell 决定，而不是依赖 sidebar 组件内部的样式猜测。实现上应让 `WorkspaceSidebar` 在有效折叠状态下退出主布局，但保留实例状态。

选择理由：

- shell 已经拥有 `effectiveWorkspaceSidebarCollapsed` 的最终判定权。
- 折叠影响的是整体布局宽度，而不是仅 sidebar 自身外观。
- 由 shell 控制可以确保 splitter、expand button 和 sidebar 同步切换。

备选方案：

- 仅在 `WorkspaceSidebar` 内增加 `.workspace-sidebar--collapsed` 样式：不合适，因为无法统一处理 layout gap、splitter 和 expand button 的协同。
- 用 `v-if` 销毁侧栏：能隐藏布局，但会丢失文件树展开态、局部菜单态和重命名输入态，用户体验更差。

### 2. 使用保留实例状态的隐藏方案

折叠时应隐藏侧栏，但不应重置侧栏内部状态。因此采用保留组件实例的方式，而不是卸载重建。这样用户在自动折叠或手动折叠后重新展开时，仍能看到之前展开的树节点与局部交互上下文。

选择理由：

- 侧栏内部已有树展开、上下文新建菜单、重命名输入等本地状态。
- 折叠应被视为视图可见性切换，而不是上下文重置。

备选方案：

- 每次折叠都重建组件：实现更简单，但会让重新展开变成“状态丢失”的操作，不符合工作台连续性。

### 3. 明确三态优先级：手动折叠、自动折叠、自动折叠下的手动重开

保留现有三态模型，但明确它的判定含义：

- 手动折叠：始终隐藏侧栏，直到用户再次主动展开。
- 自动折叠：在宽度不足时默认隐藏侧栏。
- 自动折叠下的手动重开：允许用户在受限宽度下临时重新看到侧栏，但不改变“当前处于自动折叠环境”的事实。

这要求实现继续由单一“有效折叠态”驱动最终显示，同时确保用户主动折叠会清除“手动重开覆盖”。

### 4. 侧栏顶部 chrome 在受限宽度下需要紧凑化，而不是继续沿用桌面比例

当前侧栏顶部由分段控件和单独的收起按钮组成，视觉上偏桌面布局。受限宽度下需要为该区域提供紧凑规则，例如按钮尺寸、内边距、间距和最小宽度的收敛，保证：

- `工作空间 / 模板` 仍然是同级主切换；
- 收起按钮仍然显式可见；
- 控件在较窄宽度下不出现明显挤压或脱离整体 chrome 的感觉。

选择理由：

- 仅修正折叠显示无法解决重新展开后的“能用但不协调”问题。
- 这是现有产品语义不变前提下的局部收口，不需要重做信息架构。

备选方案：

- 保持当前 header 不变，只修正折叠：行为修好，但窄宽度观感仍差。
- 将收起按钮并入 tabs 内部：可能更整洁，但会改变现有控件语义和交互结构，本轮不必扩大。

## Risks / Trade-offs

- [保留组件实例可能让折叠时的隐藏菜单继续留在 DOM 中] → Mitigation: 在折叠触发时同步关闭局部 dropdown / inline edit 等临时浮层，避免重新展开时出现陈旧悬浮状态。
- [受限宽度下允许手动重开会继续压缩主工作区] → Mitigation: 保持自动折叠为默认状态，只允许用户显式覆盖，并通过紧凑 header 规则减少额外拥挤。
- [继续使用多布尔状态模型会让后续维护者难以理解] → Mitigation: 在 shell 层保持单一“有效折叠态”输出，并通过测试锁定状态优先级，不在本轮扩大为新的状态机重构。
- [调整紧凑样式可能影响现有桌面视觉快照或选择器测试] → Mitigation: 将样式变更限定在侧栏语义类，补齐手动折叠和受限宽度测试，不改动无关 workbench 区域。

## Context

当前 workbench 已经具备基础的运行时错误结构化能力，但前后端契约和 UI 呈现还停留在“能显示错误”的层级，没有形成适合对话工作台的反馈模型。

现状有两个主要问题：

- `apps/agent-backend` 已能为部分模型错误和 tool 错误生成 `runtimeError.userMessage`，但仍会携带原始 `detail`；`apps/web` 的错误卡片又默认直接展示 `detail`，导致用户在对话中看到超时、请求地址、原始异常等内部文本。
- workbench 目前只有 `tool.started` 过程事件，没有“tool 失败但正在恢复 / tool 最终失败”的过程状态，因此用户无法判断当前是自动修正中还是运行已经终止。

这次变更横跨 `apps/agent-backend` 与 `apps/web`，属于典型的跨层契约收敛问题：后端需要明确输出哪些用户安全字段，前端需要明确哪些字段默认展示、哪些字段仅作为技术详情保留。

## Goals / Non-Goals

**Goals:**
- 为模型/网络错误建立稳定的用户安全提示分层：状态、面向用户的摘要、可选技术详情。
- 为 tool 执行失败建立可见的过程反馈，使 workbench 能区分“正在恢复”与“已经失败终止”。
- 扩展前端类型与流式事件消费逻辑，使前端能基于结构化字段渲染，而不是猜测原始错误字符串。
- 保持现有会话优先的 workbench 结构，不引入新的独立错误面板或全局通知中心。

**Non-Goals:**
- 不改动当前 agent 运行主流程、tool 恢复策略或 stop policy 的业务规则。
- 不新增第三方依赖，不调整 monorepo 顶层目录结构。
- 不把所有技术细节完全移除；调试信息仍可保留，但默认不直接展示在对话正文中。
- 不在本次变更中重做 workbench 全量消息卡片设计，只聚焦错误与恢复反馈。

## Decisions

### D1: 将运行时错误明确拆成“用户摘要”和“技术详情”

**选择**: 继续以 `runtimeError.userMessage` 作为默认展示文本，并把 `detail` 明确视为技术诊断字段，仅供折叠展示、日志、测试断言和调试使用。

**备选方案**:
- 前端继续直接展示 `detail`，只在个别文案上做清洗
- 后端完全删除 `detail`

**理由**:
- 直接展示 `detail` 会持续泄漏 request URL、内部异常链和英语原文，不适合作为面对普通用户的对话反馈。
- 完全删除 `detail` 会损失排障价值，也不利于测试与开发日志比对。
- 保留两层语义最适合当前工作台：默认安全，必要时可诊断。

### D2: 为 tool 失败新增显式过程事件，而不是只依赖 terminal error

**选择**: 在运行流中新增 tool 失败相关事件，至少覆盖“失败但将恢复重试”和“失败且已终止”两类状态；前端据此更新当前 assistant 消息头部和状态文本。

**备选方案**:
- 不新增事件，只在 terminal `run.completed` 时显示最终错误
- 仅通过日志侧通道记录 tool 失败

**理由**:
- 用户真正困惑的点不是“最后为什么失败”，而是“当前这一步到底是在恢复还是已经停了”。
- terminal 事件只表达终局，不能表达自动恢复中的中间态。
- 日志流不是主交互面，不能替代对话中的过程反馈。

### D3: 前端按失败阶段和终态/非终态分开渲染

**选择**: workbench 依据 `runtimeError.stage`、tool 失败事件类型以及终态状态，使用三种反馈层次：

- 非终态恢复中：更新 assistant header 和会话状态，不生成红色终态错误卡片
- 终态失败：渲染错误卡片，正文显示 `userMessage`
- 技术详情：默认折叠，仅在用户需要时查看

**备选方案**:
- 统一所有错误为同一种红色卡片
- 把恢复中失败也直接作为终态错误卡片展示

**理由**:
- 恢复中与终态失败在语义上不同，统一渲染会让用户误以为运行已结束。
- Header 更适合承载“正在调用 / 正在恢复 / 执行失败”这类状态切换，错误卡片只保留最终结论。

### D4: 扩展前端 `RuntimeError` 类型以完整接收后端语义字段

**选择**: 让 `apps/web/src/api/types.ts` 补齐前端缺失的 `failureKind`、`toolName`、`stopReason`、`normalizedCode` 等字段，与后端运行时错误类型保持兼容。

**备选方案**:
- 前端继续只消费精简字段
- 前端从原始 error message 中做字符串推断

**理由**:
- 当前前端类型裁剪掉了关键字段，会阻碍基于 stop reason 或 failure kind 的稳定渲染。
- 用原始字符串推断错误类别脆弱且难维护。

## Risks / Trade-offs

- **[事件契约扩大]** → 新增 tool 失败过程事件后，前后端类型、测试、渲染逻辑都要同步更新；通过共享清晰的事件语义和测试夹具降低漂移风险。
- **[信息隐藏过度]** → 如果默认隐藏过多技术信息，调试体验会变差；通过保留 `detail` 并提供折叠查看入口缓解。
- **[状态竞争]** → `tool.started`、tool 失败过程事件、`lifecycle.error`、`run.completed` 之间可能出现 UI 覆盖顺序问题；通过在 store 中明确“过程态优先于旧 header、终态覆盖过程态”的规则约束。
- **[中英文混杂]** → 现有后端仍有部分英文默认文案；通过集中在 runtime error builder 中统一中文用户提示来降低文案漂移。

## Migration Plan

1. 在 `apps/agent-backend` 扩展 runtime error builder 和流式事件定义，明确用户摘要与技术详情边界，并补充 tool 失败过程事件。
2. 在 `apps/web` 扩展 API 类型和 workbench store 事件处理，区分恢复中与终态失败。
3. 调整 conversation 错误卡片，只默认展示用户摘要，将技术详情改为可选披露。
4. 补充前后端测试，覆盖模型超时、tool 自动恢复、tool 终态失败和取消等路径。
5. 前端发布后无需数据迁移；旧 session 历史中若没有新增字段，仍按已有 `userMessage` / `error` 兜底展示。

## Open Questions

- tool 失败过程事件是否需要拆成 `tool.failed` 与 `tool.recovering` 两类，还是用单一事件配合 `terminal` 标志即可。
- 技术详情在对话中应采用折叠区、显式“查看详情”链接，还是仅保留在开发日志视图中；本次实现可以先保留最小可用披露方式。

## Context

当前问题集中在 `apps/web` 的 workbench 右侧工作空间侧栏。现有布局通过 `WorkbenchShell`、`WorkspaceSidebar` 及其内部 panel/tree 容器叠加了多层 `flex`、`height: 100%` 与 `overflow`，文件数量上来后，纵向滚动能力会在这些容器之间失稳，最终表现为文件窗口无法继续下拉或无法响应正常滚动输入。

这个问题已经确认为前端布局与滚动容器归属问题，而不是后端上传数量限制或工作空间文件数上限。整改必须保持现有侧栏结构、标签页、文件树交互和视觉层级基本不变，不引入新的依赖，也不修改上传、读取或打开文件的后端契约。

## Goals / Non-Goals

**Goals:**
- 让工作空间侧栏在长文件列表场景下持续保持可预期的纵向滚动能力。
- 将右侧侧栏收敛为单一纵向滚动容器模型，避免多层容器互相争夺滚动职责。
- 保持当前 workbench 的视觉结构、工作空间分组和文件打开行为不变。
- 通过前端测试固化滚动容器归属，降低样式回归风险。

**Non-Goals:**
- 不重做工作空间侧栏视觉样式，也不引入新的滚动组件或虚拟列表方案。
- 不修改后端工作空间数据结构、上传接口或文件数处理逻辑。
- 不顺带重构其他无关的侧栏交互逻辑。

## Decisions

### Decision: 将工作空间 panel 收敛为唯一纵向滚动容器

侧栏滚动职责将集中在工作空间 panel 容器上，外围壳层负责提供稳定的高度约束，内部文件树不再各自声明独立纵向滚动。这样可以让长列表只依赖一条清晰的高度链和一个滚动面。

Rationale:
- 单一滚动容器最容易保证滚轮、触控板和拖动滚动条行为一致。
- 当前问题的根因是多层 `overflow` 与 `height: 100%` 叠加，继续保留多层滚动只会让行为更脆弱。
- 该方案对视觉影响最小，只修正布局职责，不重写文件树结构。

Alternatives considered:
- 继续保留内部文件树滚动，仅微调局部高度：拒绝，因为仍然依赖脆弱的多层滚动关系，回归风险高。
- 直接引入虚拟列表或第三方滚动封装：拒绝，因为问题核心不是渲染量上限，而是现有布局滚动链失效，且这会扩大改动面。

### Decision: 在 shell 到 sidebar 的整条高度链补齐 `min-height: 0` 与 `overflow: hidden`

实现会在 `WorkbenchShell` 的 sidebar slot、`WorkspaceSidebar` 的根容器和中间 shell/panel 容器之间补齐必要的 `min-height: 0` 与 `overflow: hidden`，确保 flex 子项能够真正收缩，并把滚动留给最终 panel。

Rationale:
- flex 布局下缺少 `min-height: 0` 是长列表无法进入内部滚动的典型原因。
- 仅在 panel 上设置 `overflow-y: auto` 不足以解决问题，外围容器也必须允许内部收缩。
- 这能把滚动建立在明确的布局约束上，而不是依赖浏览器对 `height: 100%` 的偶然计算结果。

Alternatives considered:
- 只给最终 panel 增加 `overflow-y: auto`：拒绝，因为外围高度链仍可能拦截滚动。
- 通过 JS 动态计算高度：拒绝，因为问题可通过稳定 CSS 布局解决，没必要引入额外状态与维护成本。

### Decision: 用样式与组件测试共同约束滚动行为

这次修复将通过组件测试验证滚动容器的样式归属，至少确保工作空间 panel 持有纵向滚动，而文件树内部容器不再承担相同职责。

Rationale:
- 该问题本质是布局回归，缺少样式约束时很容易被后续改动重新引入。
- 纯手工验证无法持续保护 `overflow`/`min-height` 这种实现细节。

Alternatives considered:
- 只做手工验证：拒绝，因为无法长期防止同类回归。

## Risks / Trade-offs

- [滚动行为依赖多个容器的样式协同，后续局部样式调整可能再次破坏高度链] → 用测试显式约束关键容器的 `overflow` 与滚动职责。
- [长列表虽然恢复可滚动，但 DOM 体量本身不会下降] → 本次只修复可用性问题，不扩展到虚拟渲染；验收标准聚焦“可稳定浏览全部文件”。
- [测试环境不能真实复现浏览器物理滚动] → 通过样式归属断言覆盖主要回归面，并在实现阶段结合实际页面做针对性验证。

## Migration Plan

1. 调整 `WorkbenchShell` 与 `WorkspaceSidebar` 的布局样式，使侧栏建立稳定的内部滚动高度链。
2. 取消文件树内部重复的纵向滚动职责，保留 panel 作为唯一滚动容器。
3. 更新相关组件测试，验证长列表场景下的滚动容器归属。
4. 该变更仅涉及前端布局与测试，无需数据迁移；如需回滚，可直接恢复现有样式与测试基线。

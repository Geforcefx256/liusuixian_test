## Context

当前 `apps/agent-backend` 的工作区上传链路会将用户上传文件的原始字节直接写入 `uploads/`，而工作区文件打开链路会在后续读取时直接按 UTF-8 解码。这个行为对 UTF-8 文本成立，但对 GB2312、GBK、GB18030 以及带 BOM 的 UTF-16 文本会产生中文乱码，且乱码直到用户重新保存为 UTF-8 后才会消失。

这次变更跨越上传路由、工作区文件存储、工作区文件打开契约以及前端展示预期，属于一个适合先明确设计决策再实施的跨模块行为变更。用户已经确认接受上传后统一转为 UTF-8 保存，并接受后续下载返回 UTF-8 版本文件。

## Goals / Non-Goals

**Goals:**
- 在上传支持的文本文件时识别源文件编码，并统一转成 UTF-8 持久化。
- 让工作台在打开支持的中文源文件时直接看到可读文本，而不是乱码。
- 对不受支持编码或非文本内容显式报错，避免静默错误展示。
- 尽量将变更收敛在上传边界，减少对现有打开、编辑、保存、下载链路的侵入。

**Non-Goals:**
- 不实现“任意编码自动猜测”或宽泛的多语言字符集推断。
- 不保留原始上传字节副本，也不为文件增加“原始编码”持久化元数据。
- 不改变当前支持的上传文件类型集合。
- 不在本次设计中引入新的根目录结构或新的跨服务存储层。

## Decisions

### 1. 在上传边界完成编码识别与 UTF-8 归一化

上传接口在调用工作区持久化前先把上传字节解码成字符串，再统一按 UTF-8 写入 `uploads/`。这样可以把编码问题收口在单一入口，而不是在 `openWorkspaceFile`、保存、下载甚至后续 Agent 读取链路重复做条件处理。

之所以不选择“打开时再猜测编码”，是因为那会导致：
- 同一文件在不同读取路径可能出现不一致行为；
- 保存路径仍然需要决定是否回写原编码；
- 下载语义会变得混乱，用户无法知道拿到的是原始字节还是重新编码后的内容。

### 2. 支持集合明确限定为 UTF-8、UTF-8 BOM、UTF-16 BOM 和 GB18030

检测顺序固定为：
- 先检测 UTF-8 BOM、UTF-16LE BOM、UTF-16BE BOM；
- 无 BOM 时，使用严格 UTF-8 解码校验；
- UTF-8 校验失败后，按 GB18030 解码；
- 若解码结果不满足文本有效性要求，则返回上传错误。

选择 GB18030 而不是单独区分 GB2312/GBK，是因为 GB18030 可以覆盖目标中文编码家族，且 Node 22 的 `TextDecoder` 已具备相应能力。这个决策能满足当前业务范围，又避免引入新的第三方依赖或模糊的编码猜测逻辑。

### 3. 不保存原始编码元数据，工作区内部只存 UTF-8

一旦上传成功，工作区内的持久化内容就是 UTF-8 文本。后续打开、编辑、保存与下载都沿用现有 UTF-8 路径，不再区分“原文件编码”和“当前存储编码”。

不增加元数据字段的原因是：
- 当前需求目标是统一规范化，不需要可逆恢复原始编码；
- 现有文件映射结构无需升级即可承载这次行为变化；
- 可以减少迁移复杂度与后续状态分叉。

### 4. 失败策略采用显式拒绝，而不是降级展示

如果上传内容无法被支持的编码集合可靠解码为文本，上传接口必须返回结构化错误并终止持久化。系统不提供“继续按 UTF-8 读、展示乱码但允许保存”的降级路径。

这个决策符合当前工程的 debug-first 约束：编码失败应直接暴露，而不是被静默掩盖成错误展示。

### 5. 优先使用 Node 内建能力实现，暂不计划新增依赖

设计默认使用 Node 22 的 `TextDecoder` 完成 BOM 识别后的解码和严格 UTF-8 校验。只有在实施阶段证明确有无法覆盖的边界情况时，才考虑引入第三方编码库；若发生这种情况，需要先单独确认依赖版本和可用性，再修改 manifest 或 lockfile。

## Risks / Trade-offs

- [GB18030 回退可能把部分异常二进制误判为文本] → 通过文本有效性校验降低误判概率，例如检查控制字符比例、空内容边界和解码后可打印文本特征。
- [上传后下载不再保留原始字节] → 已由用户确认接受，且在 proposal 与 specs 中明确写成产品语义。
- [旧工作区中已存在的历史乱码文件不会自动修复] → 本次只保证新上传文件被规范化；已有文件仍按现有存储状态读取。
- [UTF-16 无 BOM 的文件不在支持范围内] → 这是刻意收敛的边界，避免模糊推断；若后续出现真实需求，再单独扩展。

## Migration Plan

1. 在 `apps/agent-backend` 的上传入口增加文本解码与 UTF-8 归一化逻辑。
2. 补充后端测试，覆盖 UTF-8、UTF-8 BOM、UTF-16 BOM、GB18030 与不受支持内容的上传行为。
3. 补充前端/集成测试，验证工作台打开上传文件时能直接读取可读中文内容。
4. 部署后新上传文件立即生效；历史文件不做批量迁移。
5. 如需回滚，可移除上传归一化逻辑并恢复原始字节写入行为；已归一化为 UTF-8 的文件保持 UTF-8 内容，不需要额外数据回滚。

## Open Questions

- 暂无阻塞性开放问题。实施阶段只需据此设计把“文本有效性校验”的具体规则收敛为可测试的函数行为。

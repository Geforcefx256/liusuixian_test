## Context

当前 workbench 文件上传冲突确认发生在 `apps/web/src/stores/workbenchStore.ts` 的上传链路中，冲突时直接调用浏览器原生 `window.confirm`。这使上传确认同时存在两个问题：一是视觉上脱离了 workbench 已有的浅色确认层、按钮与遮罩体系；二是 store 直接持有浏览器交互，导致上传覆盖确认无法由页面层统一接管。

这次变更只覆盖 `apps/web` 的前端交互层，不修改后端上传 API、冲突错误码或工作区数据结构，也不引入新的第三方依赖。用户已经明确要求不要改坏其他前端样式或交互，因此实现必须局部收敛在“上传同名文件冲突确认”这条链路上，并尽量复用现有 workbench 确认组件与 token。

## Goals / Non-Goals

**Goals:**
- 用 workbench 自己的确认弹层替换上传冲突时的浏览器原生确认框。
- 复用现有确认表面、遮罩、按钮和颜色 token，保持与会话历史确认体验一致。
- 让上传冲突确认展示明确的目标路径和覆盖后果，并支持取消、确认覆盖、点击遮罩关闭和 `Esc` 关闭。
- 将改动限制在上传覆盖链路，避免影响其他 workbench 模块和全局样式基线。

**Non-Goals:**
- 不重做 workbench 的全局 modal 架构，也不引入新的弹层库。
- 不修改后端文件上传接口、`UPLOAD_CONFLICT` 错误契约或工作区目录规则。
- 不顺带改造其他仍然存在的原生 `alert` 或其他工作台交互。
- 不扩展为批量冲突管理、差异预览、撤销覆盖或版本化上传能力。

## Decisions

### Decision: 上传冲突改为 workbench 居中确认弹层，而不是顶部 notice 或行内提示

上传覆盖是会阻断当前上传流程的显式决策，因此采用居中的小型确认弹层，沿用现有 workbench 遮罩和确认表面的视觉语言，而不是插入顶部通知条或右侧栏内联提示。

Rationale:
- 上传动作从对话区 `+` 入口触发，居中确认层比侧栏内联提示更符合当前操作语境。
- 冲突确认会中断当前上传队列，弹层能更清楚地表达“必须先做决定，流程才继续”。
- 复用已有的浅色确认层更容易保证视觉一致性，同时避免黑色系统框的割裂感。

Alternatives considered:
- 顶部 notice：层级过轻，用户容易错过，且需要自己再寻找后续动作。
- 保留浏览器 `window.confirm`：实现最省事，但继续破坏 workbench 的设计一致性。

### Decision: 复用现有确认表面与按钮 token，不改全局样式基线

实现优先复用现有确认表面、按钮类名和遮罩样式，只在上传覆盖场景补充最小必要的文案与上下文展示，不修改全局 `styles.css` 的按钮、颜色、圆角和阴影定义。

Rationale:
- 用户明确要求不要搞坏其他前端样式，复用既有 token 是最稳妥的路径。
- 现有 `SessionRailConfirmSurface` 已经具备所需的基础能力：标题、正文、多行说明、取消/确认按钮和默认焦点。
- 局部复用比重新设计一个上传专用 modal 更容易控制回归面。

Alternatives considered:
- 新建一整套上传专用 modal 样式：视觉可控，但会制造新的样式分叉，增加维护成本。
- 直接抽改全局确认样式：复用更彻底，但回归面更大，不符合本次保守改动目标。

### Decision: 上传冲突状态由 workbench 前端状态显式建模，并由 shell 负责渲染确认层

上传冲突不再在 store 内直接调用浏览器确认，而是改为显式记录“待确认覆盖”的冲突状态，包括目标 agent、原始 `File`、冲突路径和后续是否继续上传。Workbench shell 观察该状态并渲染确认层；用户确认或取消后，再回到上传流程继续处理。

Rationale:
- 这样可以移除 store 对浏览器原生交互的直接依赖，同时保留上传重试逻辑仍由同一条上传链路管理。
- 将确认渲染放在 shell，可以复用现有 Teleport + backdrop 结构，并保持交互关闭语义可测试。
- 显式状态比向 store 注入 confirm 回调更容易追踪，也更符合当前调试优先策略。

Alternatives considered:
- 由 store 接收一个异步确认回调：能减少状态字段，但会把 UI 依赖以函数注入方式塞回业务流程，测试和可读性都更差。
- 把整个上传流程搬到组件：会打散现有上传、刷新工作区、状态文案的集中管理。

### Decision: 多文件上传在出现冲突时按顺序暂停，并在用户决策后恢复当前队列

上传多个文件时，如果其中某个文件命中冲突，流程应暂停在该文件上，等待用户确认覆盖或取消；决策完成后，再继续处理剩余文件，而不是丢失后续文件或并发弹出多个确认层。

Rationale:
- 当前上传流程本来就是顺序遍历文件，顺序暂停/恢复能最小化实现改动。
- 一次只处理一个冲突确认，避免多个弹层堆叠或冲突状态相互覆盖。
- 这种串行决策更容易测试，也更符合用户对多文件上传的可预期体验。

Alternatives considered:
- 预先收集所有冲突再做批量确认：需要额外的数据聚合和批量覆盖语义，超出本次范围。
- 命中冲突即终止整批上传：会意外影响无冲突文件的正常上传，体验更差。

## Risks / Trade-offs

- [确认层复用会话历史组件后，上传场景文案或结构不够贴合] → 保持基础视觉复用，同时允许上传冲突场景传入文件路径与覆盖说明的定制文案。
- [上传队列暂停/恢复处理不当，导致多文件上传卡住或重复上传] → 将待确认文件与剩余队列状态显式建模，并通过测试覆盖确认/取消后的继续路径。
- [把上传冲突状态放入 workbench 状态后，局部 UI 状态边界变宽] → 仅暴露上传冲突确认所需的最小字段与明确的 confirm/cancel action，不将其他弹层语义一并推入 store。
- [复用现有确认层时改动共享组件，波及历史会话确认] → 优先保持现有 props 兼容；如果需要扩展，仅做向后兼容的小改动并补齐现有测试。

## Migration Plan

1. 在 workbench 上传流程中移除浏览器原生 `window.confirm` 路径，改为记录待确认覆盖状态。
2. 在 shell 层接入产品内确认弹层，复用现有确认表面与遮罩样式。
3. 为确认、取消、遮罩关闭、`Esc` 关闭以及多文件顺序恢复补齐前端测试。
4. 该变更只影响前端交互，无需后端迁移或数据迁移；如需回滚，可恢复原来的 `window.confirm` 路径。

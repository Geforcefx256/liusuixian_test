## Context

当前工作空间侧栏的核心信息由 `apps/web/src/stores/workbenchStore.ts` 生成分组标签，再由 `apps/web/src/components/workbench/WorkspaceSidebar.vue` 渲染标题、分组头、空状态和创建入口。现状把内部实现术语 `input` / `working`、共享范围提示“共享工作区”、以及文本按钮 `NEW` 直接暴露给用户，导致上传区与工程区的角色边界不够直观，空分组时也会出现重复的计数和空文案。

这次改动是现有前端 contract 的显示层调整，不涉及上传、创建、重命名、打开文件或树结构构建的底层数据模型。现有工作区逻辑大量依赖 `groupId: 'input' | 'working'` 做分支判断，测试也围绕这些分组标识组织，因此需要把显示文案与内部标识分离，而不是重命名底层分组。

## Goals / Non-Goals

**Goals:**
- 移除工作空间侧栏顶部独立的“共享工作区”标题行，让分组列表本身成为 workspace 面板的主体。
- 将用户可见分组名改为 `upload` 和 `project`，同时保留内部 `input` / `working` 标识。
- 在分组头前统一增加文件夹图标，强化分组语义。
- 让空分组保持干净，只显示分组名，不显示 `0` 计数或“暂无文件”。
- 将 `project` 右侧的顶层创建入口从 `NEW` 改成与文件夹行内一致的 `+` 触发器，并继续复用同一套创建菜单与行为。

**Non-Goals:**
- 不修改上传入口的位置；上传仍由对话框左下角的 composer `+` 负责。
- 不修改底层 workspace payload 的 `groupId`、上传来源、创建流程或树构建算法。
- 不新增第三方依赖，也不调整 monorepo 顶层结构。

## Decisions

### 保留内部 `groupId`，只变更用户可见标签

`input` / `working` 已经渗透到 store 计算、树节点构建、创建入口判定、测试数据和 spec 语义中。直接重命名底层标识会扩大改动面，并引入与上传、创建、选择逻辑无关的回归风险。

因此本次只修改 `group.label` 和相关 UI 文案，把用户看到的名字改成 `upload` / `project`，保留内部 `groupId` 继续驱动逻辑分支。

备选方案：
- 直接把 `groupId` 重命名成 `upload` / `project`。未采用，因为会扩大行为面和测试面，收益仅限显示层。

### 删除独立标题行，保留分组头作为唯一层级提示

当前“共享工作区”标题行与分组树之间存在重复层级：标题先说明“这是工作区”，下面再显示 `input` / `working` 分组。既然用户已经在“工作空间”tab 内，额外标题行价值有限，还会占用纵向空间。

因此直接移除这整行，包括其 hover 提示，让 `upload` / `project` 分组头成为唯一信息入口。

备选方案：
- 保留标题行但改文案。未采用，因为用户明确要求删除整行，而且仍会保留层级冗余。

### 空状态收敛到“仅显示分组名”

空分组当前同时暴露计数 `0` 与“暂无文件”，在没有内容时形成双重提示。对 workspace 分组来说，空状态并不需要重复教育；用户只要能看见 `upload` 和 `project` 两个干净分组即可。

因此空态规则改为：
- 始终显示分组名。
- 仅在该分组存在内容时显示计数。
- 仅在该分组存在内容或创建中的 inline input 时渲染树区域；空分组不额外显示“暂无文件”。

备选方案：
- 保留“暂无文件”但隐藏计数。未采用，因为仍会让空分组显得嘈杂。

### 顶层创建入口与文件夹行内入口统一成 `+`

顶部 `NEW` 入口与文件夹行内 `+` 入口当前已经复用同一套创建状态与菜单逻辑，差异主要在触发器文案和样式。继续保留 `NEW` 会使同类操作在同一侧栏里出现两套视觉语言。

因此将 `project` 顶层创建入口改成与文件夹行内相同的 `+` 样式，同时保持：
- 仍然只出现在 `working` / `project` 分组。
- 仍然打开相同的“新建文件夹 / TXT / MD / MML”菜单。
- 仍然默认在 `working` 根目录创建，除非用户从某个文件夹上下文触发。

备选方案：
- 仅把 `NEW` 改成 `+` 文案，不复用样式。未采用，因为用户明确要求复用文件夹新建列表的 `+`。

## Risks / Trade-offs

- [规格基线变化] 现有 OpenSpec 明确约束 `共享工作区`、`input` / `working` 和 `NEW` 文案 → 通过同一 change 更新 `agent-web-workbench` delta spec，确保实现与 contract 同步。
- [测试脆弱性] 现有 `WorkspaceSidebar.test.ts` 对旧文案有直接断言 → 在实施阶段同步改写断言，避免把旧 copy 当成功能行为继续锁死。
- [共享语义变弱] 去掉“共享工作区”标题后，用户不再从标题处感知跨会话共享 → 接受这个取舍，因为需求目标就是收紧侧栏层级与视觉噪音，不再把共享说明作为常驻信息。

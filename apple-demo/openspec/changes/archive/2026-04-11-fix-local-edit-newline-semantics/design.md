## Context

`apps/agent-backend` 新增的本地文件工具已经要求模型先调用 `read_file`，再将返回内容中的 `old_string` 复制给 `edit`。当前 `read_file` 会在构建 payload 时把 `\r\n` 和 `\r` 规范化为 `\n`，但 `edit` 仍在原始文件内容上直接匹配，导致 CRLF 文件的多行替换天然失败。

这个问题集中在本地工具实现内部，主要涉及 `readFile.ts`、`editFile.ts`、共享文本工具和 provider 测试。它不需要新增 API，也不需要引入第三方依赖，但需要把“读取视图”和“编辑语义”统一成一个明确契约。

## Goals / Non-Goals

**Goals:**
- 让 `edit.old_string` 可以可靠匹配 `read_file` 暴露给模型的多行文本视图。
- 在 CRLF 文件上支持多行替换、删除和 `replace_all`，避免 `old_string not found` 误报。
- 写回时尽量保留原文件的主换行风格，避免一次局部编辑把整份文件改成另一种行尾格式。
- 用自动化测试覆盖 LF 和 CRLF 的关键路径。

**Non-Goals:**
- 不修改 `local:edit` 的错误分类和 recoverable/terminal 语义。
- 不处理 `read_file` 的 `mtime` 采样时序问题。
- 不引入字节级 offset 映射或复杂 diff/patch 依赖。

## Decisions

### 1. 将 `read_file` 与 `edit` 统一到同一个 LF 规范化视图

**选择**：抽取共享的换行规范化函数，让 `read_file` 与 `edit` 都基于相同的 `\n` 视图工作。

**替代方案**：保留 `edit` 对原始文件内容的精确匹配，仅调整 `read_file` 输出。

**理由**：模型实际复制的是 `read_file` 的输出，因此工具契约必须围绕这个视图定义。统一到同一个规范化层后，`old_string` 的匹配语义和用户可见内容保持一致，问题范围也最可控。

### 2. 在编辑阶段记录并恢复原文件主换行风格

**选择**：`edit` 读取文件时识别原文件的主换行风格，在规范化视图上完成匹配和替换后，再将结果恢复为原风格写回磁盘。

**替代方案**：始终写回 LF，或保留 `read_file` 的规范化输出但允许 `edit` 只接受原始换行。

**理由**：始终写回 LF 会让 Windows 风格文件产生额外 diff；要求模型携带原始 CRLF 又会继续破坏“从 read_file 复制”的使用方式。恢复主换行风格能在兼容性和实现复杂度之间取得平衡。

### 3. 将混合换行文件视为可接受的规范化写回场景

**选择**：若文件同时含有多种换行风格，`edit` 使用检测到的主风格作为写回目标。

**替代方案**：为每一处替换维护原始 offset 映射，逐段保留混合行尾。

**理由**：字节级映射实现复杂度明显高于当前问题规模，而且当前工具已经是基于文本视图而非结构化 patch。对混合换行文件做一次风格规范化是可以接受的行为边界。

### 4. 用 provider 级回归测试锁定契约

**选择**：在 `localProvider.test.ts` 中增加 CRLF 文件读取后编辑成功、写回仍保持 CRLF、以及 LF 文件行为不变的测试。

**替代方案**：只补底层单元测试。

**理由**：review 暴露的是跨 `read_file -> provider state -> edit` 的集成语义，provider 级测试更接近真实 agent 工作流，也更能防止未来回归。

## Risks / Trade-offs

- **[混合换行文件被规范化]** → 文本替换后可能将少数非主流行尾统一成主风格；通过在设计中显式接受这一边界并用测试覆盖常见 LF/CRLF 文件降低风险。
- **[共享 helper 语义扩散]** → 修改共享文本工具可能影响现有 `read_file` 行为；通过保持 `read_file` 输出格式不变，只复用相同的规范化逻辑控制影响面。
- **[编辑后差异扩大]** → 若 `new_string` 自带 `\r\n`，写回阶段需要先统一到 `\n` 再恢复原风格，避免生成 `\r\r\n`；通过集中在单一恢复函数处理规避该类错误。

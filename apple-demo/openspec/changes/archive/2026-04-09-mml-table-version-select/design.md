## 背景

当前 MML 规则由 `web-backend` 从指定目录扫描 Excel 文件导入数据库，前端在工作台中再根据 `网元类型 + 网元版本` 加载 schema 并驱动表格视图。但就原始需求来看，现有实现有两个明显问题。

第一，用户切换到表格视图后，`网元类型` 和 `网元版本` 为空，并且不能下拉选择，导致表格视图缺少一个可直接操作的配置入口。第二，后台规则导入逻辑与新需求不一致：现有文件名解析仍依赖 `CHECK_RULE_<networkType>_<networkVersion>_MARCO_RULE.xlsx` 形式，且导入策略偏向“按 checksum 跳过未变化文件”，这与“按新命名规则覆盖重载并删除源文件”的需求不一致。

因此，这次设计必须同时解决前端交互和后台规则导入两件事，而且以后者为前者提供真实数据来源。

## 目标 / 非目标

**目标：**
- 在 MML 表格视图相关配置区域中，让 `网元类型` 和 `网元版本` 支持下拉选择。
- 让两个下拉框的默认显示值为 `请选择`。
- 让下拉数据由后台返回，并与后台当前已导入的规则数据保持一致。
- 将规则文件命名规则调整为 `CHECK_RULE_网元类型_版本.xlsx`。
- 对于同一 `网元类型 + 网元版本`，导入新文件前先删除数据库中的旧数据，再重新写入。
- 在规则文件成功导入并持久化后，删除原始 Excel 文件。

**非目标：**
- 不改变当前 MML schema 的核心数据结构和表格投影模型。
- 不引入额外的规则管理后台功能，例如人工激活、版本对比或导入历史查询。
- 不扩展为通用的目录浏览体验或模糊搜索选择体验。
- 不改动仓库顶层目录结构，也不引入新的第三方依赖。

## 关键决策

### 决策：前端下拉数据由后台提供，前端只负责展示和回填

前端不再把 `网元类型` 和 `网元版本` 当成自由输入项，而是通过后台接口获取可选项数据，并在表格视图相关配置区域中渲染为下拉框。两个下拉框的初始展示值统一为 `请选择`；只有用户完成选择后，前端才继续驱动 schema 加载与表格编辑能力。

这样可以直接满足原始需求里的“数据从后台获取”和“默认值为请选择”两个要求，也避免前端自行维护可选项列表。

备选方案：
- 前端本地写死类型和版本选项：拒绝。因为这会让前后端规则数据脱节。
- 继续保留自由输入，同时额外提示候选项：拒绝。因为这不能满足“支持下拉选”的明确需求。

### 决策：后台增加一个轻量选项接口，基于已导入规则数据返回类型和版本

为了支持下拉框，后台需要提供一个简单、稳定的选项接口，返回当前数据库中已有的 `网元类型` 以及对应的 `网元版本` 列表。这个接口只服务于下拉选项，不替代现有 schema 查询接口。

这样既能满足“数据从后台获取”，又能保持 schema 查询职责不变。前端先取选项，再按用户选择的组合取 schema。

备选方案：
- 把下拉选项直接塞进 schema 接口响应：拒绝。因为用户在完成选择前就需要候选数据。
- 让前端通过反复试探 schema 接口推断可用项：拒绝。因为这会把选择逻辑建立在失败请求上。

### 决策：规则导入逻辑按新命名规则覆盖重载，而不再依赖 checksum 跳过

后台规则导入逻辑将以 `CHECK_RULE_网元类型_版本.xlsx` 作为新的文件命名规则。扫描到文件后，先解析出 `网元类型` 和 `网元版本`，再检查数据库中是否已存在对应数据；若存在，则先删除旧数据，再导入当前文件内容。

这是对原始需求的直接实现，而不是现有“同 checksum 可跳过”的延续。需求要的是“有同网元同版本数据就先删再重载”，因此文档不再把 checksum 跳过作为本次设计目标。

备选方案：
- 保留现有 `_MARCO_RULE` 命名规则兼容：本次设计不作为主目标，除非实现阶段明确要求兼容旧文件。
- 继续以 checksum 是否变化决定是否跳过导入：拒绝。因为与原始需求中的覆盖式重载要求不一致。

### 决策：只在成功持久化后删除源 Excel 文件

删除原始 Excel 文件是原始需求的一部分，但删除动作只能发生在文件内容已经成功解析并写入数据库之后。如果导入失败，源文件必须保留，以便排查和重试。

这样既满足“导入后删除原文件”的需求，也避免因中途失败导致规则文件和数据库同时丢失。

备选方案：
- 扫描到文件后立即删除：拒绝。因为失败时无法恢复。
- 永不删除源文件：拒绝。因为这与原始需求明确冲突。

## 风险 / 权衡

- [删除源 Excel 文件属于不可逆动作] → 仅在数据库保存成功后删除，并记录清晰日志。
- [覆盖式重载会抛弃旧版本同组合数据] → 这是原始需求要求的行为，测试中要明确验证“先删后导”。
- [前端下拉依赖后台选项接口] → 通过空数据和异常场景测试，确保下拉无数据时仍能明确展示 `请选择` 与不可编辑状态。
- [新命名规则可能与现有历史文件不兼容] → 本次设计以新规则为准，若需要兼容旧规则，应作为后续明确需求单独提出。

## 迁移计划

这次变更不涉及数据库结构迁移，主要是逻辑调整：

1. 修改后台文件名解析与导入流程，按新命名规则执行同组合覆盖式重载，并在成功后删除源文件。
2. 增加前端下拉选项所需的后台接口，返回当前网元类型和版本候选数据。
3. 前端将 `网元类型` 和 `网元版本` 改为下拉框，默认显示 `请选择`，并在选择后继续现有 schema/table 流程。
4. 补充前后端测试，覆盖新命名规则、重载删除逻辑、源文件删除行为，以及表格视图下拉交互。

回滚方式也较明确：恢复旧文件命名解析和导入策略，并将前端选择器还原为当前输入方式。由于底层 schema 消费路径不变，风险主要集中在导入和交互层。

## 开放问题

- 新命名规则是否需要在实现阶段同时兼容旧的 `_MARCO_RULE` 文件名？原始需求未要求兼容，当前设计默认不兼容。
- 后台返回的下拉选项是否需要排序规则，例如按网元类型字母序、版本自然序？当前设计建议使用稳定排序并用测试固定，但不把具体排序策略作为本次需求核心。

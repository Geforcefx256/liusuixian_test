## Context

当前历史会话侧栏已经具备两类危险操作：单条会话删除和“清空历史会话”。这两条路径都依赖浏览器原生 `window.confirm`，因此确认样式脱离了当前 workbench 的视觉 token、按钮体系和 hover/focus 驱动的侧栏上下文。

这次变更只覆盖 `apps/web` 的前端交互层，不改动后端删除/清空 API，也不引入新的第三方依赖。当前实现里最重要的约束有三点：
- 历史侧栏本身是 hover/focus 触发展开与关闭的覆盖层，删除确认不能破坏这套进入/离开逻辑。
- 单条删除发生在具体会话行内，用户需要明确知道“删的是哪一条”。
- “清空历史会话”风险明显更高，虽然希望与单条删除保持统一风格，但层级必须更正式。

## Goals / Non-Goals

**Goals:**
- 用产品内确认交互替换历史会话里的浏览器原生确认框。
- 让单条删除使用贴近会话项的轻量确认浮层，保留会话标题上下文和短文案。
- 让“清空历史会话”使用同一视觉语言下更正式的小型确认框，明确当前会话不受影响。
- 为两类确认补齐一致的取消、确认、点击外部关闭和 `Esc` 关闭语义，并保持可测试。

**Non-Goals:**
- 不修改会话删除、批量清空历史的后端契约与数据边界。
- 不把历史侧栏整体改造成常驻宽侧栏，也不重做其 hover 展开机制。
- 不新增撤销删除、软删除、归档替代删除或多选批量管理能力。
- 不为了这次交互改造引入通用弹层库或新的设计系统依赖。

## Decisions

### Decision: 单条删除使用锚定在会话项附近的轻量确认浮层

单条删除不再弹出浏览器系统框，而是在历史侧栏内部渲染一个贴近目标会话项的轻量确认浮层。浮层内容只保留必要信息：动作标题、会话标题上下文、不可恢复提示，以及“取消”/“删除会话”两个动作。

Rationale:
- 单条删除是局部、高频、上下文很强的危险操作，轻量浮层比居中确认框更符合侧栏列表的心智模型。
- 保留会话标题可以直接告诉用户“正在删除哪条会话”，比原生确认框更清晰。
- 这种处理方式更接近现代 AI/chat 产品的侧栏危险确认气质：轻、近、克制。

Alternatives considered:
- 居中确认框：层级过重，会把用户从侧栏上下文里抽离。
- 保留 `window.confirm`：实现最简单，但继续破坏产品内一致性，也无法细化焦点与视觉层级。

### Decision: 清空历史使用同风格但更正式的小型确认框

“清空历史会话”继续要求显式确认，但不与单条删除共用同一种轻浮层。它将使用更正式的小型确认框，以更清楚地表达作用域、当前会话排除规则和不可恢复性，同时视觉 token 与按钮体系仍与单条删除保持一致。

Rationale:
- 该操作影响的是当前助手下的多条历史记录，风险和影响面都高于单条删除。
- 如果也做成轻浮层，会削弱风险层级，用户更难感知这是批量不可恢复操作。
- 视觉语言统一但层级区分，可以同时满足一致性和风险表达。

Alternatives considered:
- 两个动作都用轻浮层：单条删除合适，但批量清空层级偏轻。
- 两个动作都用正式 dialog：会让单条删除显得过重，降低侧栏操作流畅度。

### Decision: 确认状态留在 SessionRail 本地，而不是引入全局 modal 管理

确认交互的瞬时状态、目标会话标识、当前打开的确认层级都由 `SessionRail` 本地持有。组件继续只向上层发出现有的删除/清空事件，后端 contract 与 store 流程保持不变。

Rationale:
- 这两类确认完全属于历史侧栏局部交互，没有必要抽成全局 modal 状态。
- 本地化状态更容易与现有的 hover/focus/菜单关闭逻辑整合，也更容易覆盖组件测试。
- 可以避免把视觉细节和瞬时 UI 状态推入 store。

Alternatives considered:
- 引入全局确认服务：会增加抽象和状态分发成本，当前范围不值得。
- 把确认状态放进 store：会让业务 store 持有过多局部 UI 细节。

### Decision: 明确可关闭语义，但不添加静默降级或兼容双路径

这次变更会直接移除 `window.confirm` 在历史会话这两个入口中的主路径，统一使用产品内确认层。关闭语义明确为：点击取消、点击外部、`Esc` 关闭；确认后才发出危险操作事件。

Rationale:
- 项目约束要求避免静默 fallback。既然要改造，就应把原生框完全退出主路径，而不是留双路径兼容。
- 显式关闭语义使组件行为可预测、可测试，也方便后续维护。

Alternatives considered:
- 保留“原生 confirm 作为兜底”：会留下两套行为分支，增加调试成本，也违背当前调试优先策略。

## Risks / Trade-offs

- [轻量确认浮层与 hover 关闭区域冲突，导致鼠标移入确认层时侧栏提前收起] → 将浮层渲染在历史侧栏展开面板的稳定交互区域内，并把确认层命中区域纳入关闭判断。
- [侧栏本身已使用 `role="dialog"`，再叠加确认层容易造成语义和焦点混乱] → 单条删除采用轻量确认浮层语义，不强行引入第二个 modal dialog；批量清空才使用正式确认框并补齐焦点返回。
- [点击外部关闭、菜单关闭、确认关闭之间出现状态竞争] → 将历史菜单、单条删除确认、批量清空确认整理为互斥状态，始终只允许一种危险确认表面处于打开状态。
- [视觉上过度模仿 OpenAI 而与现有蓝灰 token 体系冲突] → 参考其层级、文案克制和上下文贴近方式，但继续使用当前 workbench 的中性色、圆角和按钮 token。

## Migration Plan

1. 在 `SessionRail` 内部抽离危险操作确认状态，去除该组件对 `window.confirm` 的依赖。
2. 先落单条删除轻量确认浮层，再补“清空历史会话”的正式确认框，并保证两者互斥。
3. 更新组件测试，覆盖打开、取消、确认、`Esc`、点击外部关闭以及作用域文案。
4. 该变更仅影响前端交互，无需后端迁移或数据迁移；回滚时可恢复原有 `window.confirm` 路径。

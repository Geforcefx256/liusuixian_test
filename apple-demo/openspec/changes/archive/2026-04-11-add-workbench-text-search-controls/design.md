## Context

当前工作台文件壳已经支持文本编辑、保存、Markdown 预览和 MML 表格视图，但文本编辑区的高频修订操作仍然停留在“可输入、可保存”的最低能力。目标用户是业务/配置人员，主要依赖鼠标点击而不是快捷键，因此如果搜索、替换和撤销只存在于编辑器默认快捷键或隐藏能力里，实际可发现性不足。

这次变更是前端工作台内的跨组件改动：外层 `WorkspaceEditorPane` 负责工具栏和可见性，内层 `WorkspaceTextEditor` 负责调用 Monaco 编辑能力。现有 Monaco runtime 采用精简入口，设计上需要优先复用现有编辑器能力，而不是引入新的独立编辑器或外部依赖。

## Goals / Non-Goals

**Goals:**
- 为当前打开文件的文本视图提供清晰可发现的搜索入口。
- 让搜索条按需展开，保持默认工具栏简洁，只常驻 `搜索`、`保存` 和 `更多`。
- 在搜索条中承载替换能力，只支持当前匹配替换和全部替换当前文件匹配项。
- 通过 `更多` 菜单为文本编辑提供可点击的 `撤销` 动作，不依赖快捷键。
- 将行为严格限制在当前打开文件的文本编辑上下文，不改变表格视图或工作区级搜索模型。

**Non-Goals:**
- 不提供工作区范围、文件树范围或跨文件搜索。
- 不实现重做、跳转到行、高级匹配选项、自动补全搜索建议或快捷键教学。
- 不改变现有保存语义、Markdown 预览行为或 MML 表格编辑语义。
- 不引入新的第三方依赖或替换现有 Monaco 编辑器体系。

## Decisions

### 1. 工具栏保持精简常驻，搜索条按需展开

在外层工作台工具栏仅常驻展示 `搜索`、`保存` 和 `更多`。点击 `搜索` 后，在文本编辑区域上方展开一条内嵌搜索条；关闭后恢复紧凑布局。

原因：
- 目标用户更依赖可点击按钮，但同时希望界面不要显得按钮过多。
- 搜索是高频入口，值得常驻；替换和撤销则不需要默认占据同级空间。
- 内嵌搜索条比弹窗更像编辑器自身能力，也不会遮挡正文。

备选方案：
- 让 `搜索 / 替换 / 撤销` 全部常驻：可发现性更强，但会让顶部快速变得拥挤。
- 把搜索也收进 `更多`：视觉更干净，但对鼠标用户的发现成本过高。

### 2. 替换作为搜索条内的二级展开，而不是工具栏独立按钮

搜索条默认只显示搜索输入与上下一个定位动作。用户点击 `替换` 后，再在同一条区域内展开替换输入以及 `替换当前`、`全部替换`。

原因：
- 替换天然依赖搜索上下文，跟搜索放在一起最符合用户心智。
- 二级展开可以压缩默认高度，避免第一次进入文本视图时信息噪音过大。
- 当前第一版范围较小，不需要做独立替换面板或弹窗工作流。

备选方案：
- 工具栏单独放 `替换`：会增加常驻密度，而且与搜索状态脱节。
- 用弹窗承载替换：会打断阅读和编辑，不适合连续修订。

### 3. 撤销放入 `更多` 菜单，且仅作用于文本编辑上下文

`更多` 菜单第一版只需要显式提供 `撤销`。该动作只对当前文本编辑器实例负责，不扩展到表格视图或跨视图统一历史。

原因：
- 用户已经明确希望有点击入口，但同时又偏好界面干净。
- 撤销属于有用但非主路径动作，放进 `更多` 可以平衡可见性和密度。
- 将撤销限定为文本编辑上下文，能避免把表格编辑历史与文本编辑历史混合，降低状态模型复杂度。

备选方案：
- 常驻显示 `撤销`：更直接，但会让精简工具栏失去约束。
- 统一撤销文本和表格修改：实现复杂，且超出本次范围。

### 4. 优先复用 Monaco 能力，但以产品语义为准包装对外接口

`WorkspaceTextEditor` 负责承接查找、替换、撤销等编辑动作，`WorkspaceEditorPane` 只负责展示按钮、展开状态和调用时机。若现有 Monaco 精简运行时未自动带出所需查找能力，应在现有 runtime 内补齐对应贡献加载，而不是新增另一套查找实现或更换编辑器。

原因：
- 当前系统已经围绕 Monaco 建立语言模式、诊断和 MML 辅助能力，继续复用最稳。
- 将产品交互状态放在外层、编辑器命令放在内层，职责更清晰，也更利于测试。
- 避免引入新的第三方依赖，符合仓库依赖治理要求。

备选方案：
- 在外层自行实现文本搜索和替换逻辑：会复制编辑器状态管理，容易与 Monaco 光标/选区脱节。
- 替换 Monaco 或新增外部插件：超出当前问题范围，并触发依赖确认成本。

## Risks / Trade-offs

- [Monaco 精简入口未自动提供查找替换 UI 能力] → 优先在 runtime 中补齐所需 contribution；如果某项能力需要额外接线，也保持接口仍由 `WorkspaceTextEditor` 封装。
- [搜索条展开后压缩编辑区可视高度] → 保持搜索条为单层紧凑布局，只有用户主动点 `替换` 时才增加第二行。
- [用户误以为搜索覆盖整个工作区] → 在交互文案和行为上都限制为当前打开文件，不在工具栏或搜索条中出现“全局/工作区”暗示。
- [更多菜单后续动作增加导致语义混乱] → 第一版仅放 `撤销`；后续若再加动作，继续按“编辑动作”维度归组，避免混入文件级操作。

## Migration Plan

该变更仅影响前端工作台交互，无需数据迁移或后端协议迁移。发布时可与现有编辑壳并行替换；若需要回滚，只需移除搜索条、`更多 -> 撤销` 入口以及对应编辑器接线，恢复当前文本编辑工具栏布局即可。

## Open Questions

- None. 当前范围已经明确限定为当前文件、按需展开搜索条、替换为搜索条内二级展开、`撤销` 放入 `更多`，且不包含重做与跳转到行。

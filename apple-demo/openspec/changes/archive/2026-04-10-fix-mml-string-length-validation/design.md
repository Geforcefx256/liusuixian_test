## Context

当前 MML 规则链路分成两段共享语义的实现：

1. `apps/web-backend` 在导入 Excel 规则时，把 `最大长度`、`最小长度`、`长度` 三列写入参数级 `lengthConstraints`。
2. `apps/web` 在 Monaco 文本诊断和 workbook 表格编辑中共享同一套参数值校验逻辑，并会同时消费 `minLength`、`maxLength`、`exactLength`。

这导致 Excel 中 `类型=字符串` 的参数只要带有 `长度` 列，就会在前端被解释成“必须等于某个长度”。问题不是单点 UI 表现，而是导入语义和消费语义共同造成的共享行为。

本次变更横跨 `web-backend` 与 `apps/web` 两个模块，但不引入新依赖，也不调整现有 schema 路由、存储模型或顶层目录结构。

## Goals / Non-Goals

**Goals:**
- 将字符串参数的长度语义收敛为 `最小长度 + 最大长度` 边界校验。
- 保证新导入规则不再把 `长度` 列写成字符串的精确长度约束。
- 兼容已导入的历史 schema 数据，避免在重新导入前继续触发错误的字符串等长校验。
- 让文本编辑和表格编辑继续共享同一套字符串长度校验语义。

**Non-Goals:**
- 不移除 schema 结构中的 `exactLength` 字段。
- 不改变非字符串参数对 `长度` 列的既有解释。
- 不引入数据库迁移或批量修复已有存量数据的离线脚本。
- 不调整 `/web/api/mml/schema` 的路由形式或前端加载时机。

## Decisions

1. 在导入层按参数类型区分长度语义。
- Decision: `web-backend` 仅对 `类型=字符串` 的参数忽略 Excel `长度` 列，不再为其写入 `exactLength`；`最小长度` 与 `最大长度` 继续保留。
- Rationale: 问题源头是字符串参数对 `长度` 列的错误解释。只在导入层做类型化收口，既能修正新数据，又不会无意扩大到其他参数类型。
- Alternative considered: 完全删除 `exactLength` 字段或停止导入所有类型的 `长度` 列；拒绝，因为这会扩大影响面，并可能破坏其他类型仍依赖精确长度的既有语义。

2. 在前端校验层增加字符串兼容保护。
- Decision: `apps/web` 在字符串参数校验时显式忽略 `exactLength`，只使用 `minLength` 和 `maxLength`。
- Rationale: 仅修改后端导入不能覆盖已入库的历史规则；前端兼容保护可以让文本视图与表格视图立即停止误报。
- Alternative considered: 要求所有环境先重新导入规则后再发布前端；拒绝，因为上线顺序不可控，而且会让旧数据在过渡期内继续产生错误诊断。

3. 继续复用当前共享校验链路，而不是拆分文本/表格两套实现。
- Decision: 保持 `validateParameterValue()` 作为文本与表格的统一参数校验入口，仅修正其中的字符串长度分支。
- Rationale: 当前两条编辑路径共享同一 schema 语义是既有设计，改动应聚焦在规则解释而不是引入新的分叉。
- Alternative considered: 为表格编辑单独定制字符串校验逻辑；拒绝，因为这会让同一 schema 在文本和表格中产生不一致行为。

4. 用测试区分“新导入语义”和“历史数据兼容”。
- Decision: 后端测试覆盖字符串导入时 `exactLength` 被抑制；前端测试覆盖字符串 schema 即便带有历史 `exactLength` 也只按最小/最大长度生效。
- Rationale: 这两个行为分别保护未来数据和当前存量数据，缺一不可。

## Risks / Trade-offs

- [Risk] 某些规则文件可能实际依赖字符串 `长度` 列表达精确长度 → Mitigation: 变更范围只针对 `类型=字符串`，并通过 OpenSpec 明确新的规范语义；若后续发现特例，应单独讨论新的参数类型或显式字段，而不是继续复用歧义列。
- [Risk] 前端忽略历史 `exactLength` 后，旧数据表现与数据库中存储值不完全一致 → Mitigation: 这是有意的兼容保护；通过重新导入规则可逐步让存储语义与运行时语义重新一致。
- [Trade-off] 保留 `exactLength` 字段会让 schema 结构暂时存在“字段还在、但字符串不用”的情形，换来更小的改动面和更低的回归风险。

## Migration Plan

- 发布顺序建议先前端、后后端：
  - 前端上线后，历史已入库规则立即停止触发字符串等长误报。
  - 后端上线并重新导入规则后，新生成 schema 将不再为字符串参数写入 `exactLength`。
- 本次不要求数据库 schema 迁移。
- 若回滚：
  - 前端可回滚到上一版本，恢复旧校验行为。
  - 后端可回滚到上一版本并重新导入规则，恢复旧导入语义。

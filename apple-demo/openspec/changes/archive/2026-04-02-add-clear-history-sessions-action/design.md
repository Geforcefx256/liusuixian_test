## Context

当前工作台历史栏已经支持悬浮展开、搜索历史会话、单条删除会话和返回空白草稿态。会话列表以 `activeAgentId` 为作用域加载，前端与后端都将“会话”视为完整持久化实体，而不是可独立治理的纯消息集合。后端现有删除契约是单条 `session` 删除，并会同时移除该 session 的 messages、summaries、plans 和 interactions。

这次变更同时跨越前端交互层和后端持久化契约：
- 前端需要在历史栏里增加一个低频危险操作入口，并清楚表达“只清空历史，不动当前会话”的范围。
- 后端需要提供与该语义一致的批量删除能力，避免前端通过循环调用单条删除接口拼装出不一致的半成功状态。

## Goals / Non-Goals

**Goals:**
- 允许用户一键清空当前助手下除当前会话之外的全部历史会话。
- 将该操作限制在历史栏语义域内，避免与“新建会话”等主操作同权重竞争。
- 保证批量清空后的前后端状态一致，包括历史列表、预览元数据和该批量删除目标会话的持久化关联状态。
- 在无可清空目标或当前会话正在运行时，为用户提供明确禁用反馈。

**Non-Goals:**
- 不提供跨助手的全局历史清空能力。
- 不提供多选会话、分组清理或按筛选结果局部清理。
- 不清空当前会话并保留其壳状态。
- 不物理删除 `user + agent` 工作区文件，也不把工作区生命周期绑定到历史清空操作上。

## Decisions

### Decision: 批量清空入口放在历史面板头部的更多菜单中

历史清理属于低频危险管理动作，不应与“新建会话”并列为主操作按钮。入口放在历史面板展开态头部的 `⋯` 更多菜单中，可以保持动作与历史列表强关联，同时降低误触概率。

Rejected alternatives:
- 将“清空历史会话”直接作为头部常驻红色按钮：曝光过高，容易与主流程动作竞争。
- 将入口放到对话区或全局设置中：作用域不直观，用户不容易理解它只影响当前助手的历史会话。

### Decision: 清空范围固定为“当前助手下排除当前会话的 persisted sessions”

前端当前会话承载实时消息、pending interaction、运行状态和会话级派生信息。若把当前会话也纳入清空范围，会把“清理历史”和“中断当前工作上下文”混成一个动作。为保持用户心智稳定，批量清空只删除 `activeAgentId` 下、且 `sessionId !== activeSessionId` 的 persisted sessions。

Rejected alternatives:
- 包含当前会话一起删除：需要额外处理正在查看中的消息流、pending question 和状态回收，交互断裂感强。
- 保留 session 壳但只清空 messages：会让 title、preview、planState、summary 和 interactions 的归属语义变得模糊，设计与实现成本都更高。

### Decision: 后端提供显式批量清空历史接口，而不是由前端循环调用单条删除

批量危险操作需要单次请求内完成目标选择、持久化删除和结果回传，避免前端循环调用单条删除接口时出现部分成功、部分失败和列表闪烁。后端将提供一个显式的 session-history clear 契约，按 `userId + agentId` 作用域删除所有历史会话，并排除传入的当前 `sessionId`。

该接口应返回已删除数量和被保留的 `excludedSessionId`，以便前端统一更新提示文案与列表状态。

Rejected alternatives:
- 前端遍历 `sessions[]` 调用现有单条 delete：失败恢复差，网络往返多，且无法提供原子化语义。
- 复用单条 delete 并允许一次传多个 `sessionId`：会把“显式动作”扭曲成“多值 delete”，可读性和后续治理都更差。

### Decision: 运行中阻止优先在前端显式治理，后端保持范围保护

当前产品里运行态主要由工作台前端感知。对于“清空历史会话”来说，只要范围强制排除当前会话，用户最关心的风险就是误删正在进行的上下文，因此前端应在 `isRunning` 为真时直接禁用该菜单项并给出原因。后端仍然通过 `excludeSessionId` 保证当前会话不会被批量删除，但不额外引入新的运行态持久化或全局会话锁。

Rejected alternatives:
- 后端新增全局运行锁并拒绝所有批量清空：需要扩展运行态真相源，超出本次需求。
- 只靠确认文案，不做禁用：对危险操作反馈不够明确。

### Decision: 工作区文件不随历史清空删除

当前工作区是按 `user + agent` 作用域存在，不依附某条 session。历史清空只作用于会话持久化记录，不改变工作区文件与侧边栏的现有生命周期模型。

## Risks / Trade-offs

- [Risk] 用户可能误解“清空历史会话”等于删除所有消息，包括当前会话。 → Mitigation: 按钮文案与确认文案都明确写出“当前会话不会受影响”。
- [Risk] 历史列表为空时仍展示危险入口会增加认知负担。 → Mitigation: 无可清空目标时将菜单项置灰，并在文案中表达“暂无可清空历史”。
- [Risk] 前端局部禁用而后端不感知运行态，外部客户端仍可能在其他界面发起同类请求。 → Mitigation: 后端至少强制排除传入的当前 `sessionId`，确保主工作上下文不被该接口误删。
- [Trade-off] 本次不做多选批量删除，无法覆盖更细粒度的历史整理场景。 → Mitigation: 先交付明确的一键清空路径，后续若确有高频局部整理需求，再单独设计多选批量操作。

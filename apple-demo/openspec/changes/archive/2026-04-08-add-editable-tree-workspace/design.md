## Context

当前工作区实现把上传文件视为只读 `参考资料`，把运行时产物视为可写 `生成结果`，并以“每组内平铺文件列表”的方式渲染右侧文件区。这套模型与当前需求冲突：用户希望 `input` 表示“带进来的输入材料”而不是“只读资料”，希望这些输入材料可以编辑并保存，还希望 `working` 成为真正的树状工作目录，支持文件夹、新建空白文档与文件夹重命名。

这次变更会同时修改 `apps/web` 与 `apps/agent-backend`。前端需要把工作区从平铺文件列表升级为树状导航，并把自动保存、`NEW` 入口与文件夹交互收敛到右侧工作区；后端需要把上传保存 contract 从“只保留 basename 的只读上传文件”改成“可保留相对路径、可原地保存的 input 文件”，同时补上 `working` 文件夹与空白文件创建的元数据与存储能力。

## Goals / Non-Goals

**Goals:**
- 将工作区用户可见分组改为 `input` 与 `working`。
- 让上传进入 `input` 的文件可打开、可编辑、可保存，并保留在 `input`。
- 让 `input` 与 `working` 都能在前端以树状结构展示层级路径。
- 让对话框 `+` 继续作为唯一上传入口，并支持把目录层级带入 `input`。
- 在右侧工作区为 `working` 提供专用 `NEW` 入口，支持新建文件夹、TXT、MD、MML。
- 支持 `working` 文件夹重命名。
- 首版自动保存采用失焦、切换文件前、关闭文件前保存。

**Non-Goals:**
- 不引入拖拽移动、跨文件夹移动文件、批量操作。
- 不把 `NEW` 合并回对话框 `+` 菜单。
- 不新增新的第三方依赖或顶层目录结构调整。
- 不在本次变更中引入版本历史、回收站、协同编辑或目录级权限模型。

## Decisions

### Decision: 保留 `source` 与 `group` 解耦，用户可见分组改为 `input` / `working`

后端内部继续保留文件来源概念，用来区分上传文件与运行时/手工创建文件；但用户可见工作区分组切换为 `input` 与 `working`。上传文件默认进入 `input`，运行时产物与手工新建文件/文件夹进入 `working`。

Rationale:
- `input` 的新语义是“用户带进来的输入材料”，不再与只读绑定。
- `working` 是工作目录语义，比 `生成结果` 更适合承载空白文件、新建文件夹与长期编辑中的文档。
- 保留内部来源概念可以减少与运行时 artifact 注册逻辑的耦合改动。

Alternatives considered:
- 直接把内部 `source` 也改成 `input` / `working`：rejected，因为会把“来源”和“用户可见角色”混成同一个概念，扩大运行时改动面。

### Decision: 上传保留相对路径，目录树在前端由路径段构建

对话框 `+` 仍是唯一上传入口，但上传 flow 需要支持把文件的相对路径带到后端。前端在目录选择或目录拖拽时，逐个上传文件，并为每个文件携带其相对路径；后端按该相对路径写入 `uploads/` 并在 workspace metadata 中保留它。前端根据 workspace entry 的相对路径构建 `input` / `working` 树。

Rationale:
- 现有单文件上传 contract 容易沿用，目录上传只需要给单个文件增加可选相对路径字段。
- 前端用路径段构树，避免后端必须维护一份专用树结构 payload。
- `input` 中文件夹通常来自上传路径而不是手工创建，因此不需要为 `input` 设计额外目录创建入口。

Alternatives considered:
- 新增一套后端直接返回完整树的 API：rejected，因为会让 metadata shaping 与 UI 表示过度耦合。
- 保持上传只存 basename：rejected，因为无法满足 `input` 层级浏览与打开文件夹内文件的要求。

### Decision: `working` 文件夹使用显式工作区节点，支持空文件夹与重命名

仅靠文件相对路径推导目录树不足以表示空文件夹，也无法可靠支持文件夹重命名。因此后端工作区元数据需要引入可持久化的文件夹节点，至少覆盖 `working`。前端树可以统一消费“文件节点 + 文件夹节点”，但 `input` 的文件夹节点可继续由路径推导生成。

Rationale:
- `NEW` 需要支持新建空文件夹。
- 文件夹重命名需要一个稳定身份，不能只靠子文件路径批量推断。
- 允许 `working` 目录树同时包含显式空目录和隐式父目录。

Alternatives considered:
- 只根据文件路径推断所有目录：rejected，因为空目录无法保留，文件夹重命名会退化成隐式批量移动。
- 为 `input` 与 `working` 都引入完全相同的显式目录持久化模型：deferred，因为 `input` 目录主要由上传路径产生，首版无需为其增加手工目录管理。

### Decision: `NEW` 仅作用于 `working`，且在当前文件夹上下文中创建

右侧工作区增加专用 `NEW` 按钮，但它只在 `working` 范围生效。创建目标目录默认为当前选中的 `working` 文件夹；如果当前选中的是 `working` 文件，则落到该文件所在目录；如果没有明确上下文，则落到 `working` 根。

Rationale:
- 这与用户确认一致：上传和新建是两类不同入口，`input` 不支持手工造内容。
- 上下文创建能减少额外的路径选择对话框。
- `working` 更适合作为工作目录，承载手工文档和空文件夹。

Alternatives considered:
- 允许在 `input` 下 `NEW`：rejected，因为会把“输入材料”与“工作目录”再次混淆。
- 让 `NEW` 始终在根目录创建：rejected，因为多层目录下可用性较差。

### Decision: 自动保存首版只覆盖失焦、切换文件前、关闭文件前

首版不做输入停顿后的 debounce 自动保存，而是在编辑器失焦、切换活动文件前、关闭活动文件前触发保存。继续处理当前文件的现有流程保持“先保存再继续处理”。

Rationale:
- 当前 store 已有显式 `saveWorkspaceFile()` 路径和“继续处理前保存”的收口，复用成本最低。
- 不引入 debounce 可以减少并发保存、保存状态竞态和定时器管理复杂度。
- 这已经能满足“自动保存而不是要求手动点保存”的首版体验。

Alternatives considered:
- 输入后延迟自动保存：rejected for v1，因为需要处理更多保存竞态与高频写入问题。

### Decision: 文件与文件夹重命名都保持“同目录内 basename 变更”边界

文件重命名继续沿用“同目录内 basename 变更、不可改扩展名、不可改父目录”的 v1 边界。新增的文件夹重命名也采用相同边界：只允许修改当前文件夹名，不支持跨目录移动。

Rationale:
- 这与当前文件重命名边界一致，便于用户理解。
- 避免把“重命名”升级成路径移动或目录重组能力。
- 可以在后端显式拒绝超边界请求，符合仓库当前的显式失败治理基线。

Alternatives considered:
- 文件夹重命名顺便支持移动：rejected，因为会演变成目录树重排。

## Risks / Trade-offs

- [上传相对路径需要浏览器端额外携带 metadata] → 通过沿用单文件上传并增加可选相对路径字段，避免引入全新批量上传协议。
- [树节点既有显式文件夹也有路径推导文件夹，前端构树逻辑会更复杂] → 统一使用规范化路径和稳定 node id，在 store 内集中构树。
- [上传文件改成可写后，`input` 不再等于只读] → 在 spec 中明确 `input` 的新语义是“输入材料”而不是“只读资料”。
- [自动保存可能在切换或关闭时暴露真实保存失败] → 保持失败显式可见，并阻止依赖该保存结果的后续动作继续执行。

## Migration Plan

1. 先更新 OpenSpec，替换工作区分组、上传可写、树状展示、`NEW` 与文件夹 contract。
2. 后端扩展 workspace metadata、上传相对路径保存、上传文件写回、`working` 文件/文件夹新建与文件夹重命名能力。
3. 前端重构工作区 store 与 sidebar，把平铺列表升级成 `input` / `working` 树，并接入 `NEW` 与自动保存。
4. 回归上传、目录浏览、打开、自动保存、`working` 新建、文件夹重命名、下载、删除与现有运行时产物展示。

Rollback strategy:
- 如果树状展示或目录上传存在问题，可以先回退到平铺文件展示和 basename 上传，但这会同时回退 `input` 层级浏览要求。
- 如果自动保存在首版验证中暴露过多编辑器时序问题，可以暂时回退到“切换/关闭前保存”，但不应恢复上传文件只读模型。

## Open Questions

- 浏览器目录选择是否需要同时支持显式“上传文件夹”动作，还是只通过拖拽目录与 `webkitdirectory` 能力隐式支持。
- `working` 文件夹删除是否纳入本次，还是仅支持新建与重命名。

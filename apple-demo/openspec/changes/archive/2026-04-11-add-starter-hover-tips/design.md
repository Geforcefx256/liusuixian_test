## Context

当前空会话工作台已经把快速开始 starter 卡片做成三列信息密集布局，并通过点击技能名展开摘要、再点击“开始使用”发送 `starterPrompt`。这条路径适合快速扫描，但当快速开始摘要文本偏长时，用户无法在不破坏卡片高度的前提下读到完整摘要。

现有实现与治理模型已经区分了两类文本：

- starter 面使用的短摘要：`starterSummaryText`
- 治理后的完整说明：`skill.description`

因此这次 change 不需要再引入新的治理字段，也不需要改变“开始使用”的主操作路径，而是需要给桌面端 starter 卡片增加一个轻量、可控、不会打乱布局的说明披露面。

## Goals / Non-Goals

**Goals:**
- 为空会话快速开始中的已展开技能提供快速开始摘要的补充阅读入口。
- 保持 starter 卡片三列布局的稳定高度与扫读效率。
- 将触发入口限制为显式的 `i` 信息按钮，避免用户误以为整卡 hover 都会弹出说明。
- 固定第一版说明卡位置策略和文本展示边界，优先稳定落地与可测试性。
- 不新增治理字段、不新增第三方依赖。

**Non-Goals:**
- 不改造搜索结果区的展开模式或说明卡交互。
- 不为移动端或窄屏触控场景提供新的点击弹层设计。
- 不引入自动避让、智能翻转、箭头跟随或浮层内部滚动。
- 不调整 Skill 管理页的治理文案策略或增加描述长度约束。

## Decisions

### 1. 仅在桌面端为 `i` 信息按钮提供 hover/focus 说明卡

说明卡只由已展开技能条目中的 `i` 按钮触发，支持鼠标 hover 和键盘 focus；整张技能条目本身不作为说明卡触发面。

这样做的原因是：

- 可以向用户明确表达“这里有补充说明”，而不是让用户猜测哪些区域支持 hover。
- 保留现有点击技能名展开摘要的主路径，不把 hover 行为混进摘要展开逻辑。
- 让“摘要展开”和“摘要补充查看”形成稳定分工。

备选方案：

- 整卡 hover 触发说明卡：可发现性弱，用户很难预期热区边界。
- 点击 `i` 打开 popover：会引入额外的打开/关闭状态管理，这一轮并非必需。

### 2. 说明卡内容固定为“标题 + 快速开始摘要”，主操作仍留在原卡片

说明卡只展示：

- governed skill name
- governed `starterSummary`

不在说明卡中重复放置：

- `开始使用`
- 额外按钮
- 搜索或跳转动作

这样可以避免说明卡从“提示”膨胀成“次级操作面板”，也能保持原 starter 卡片的执行路径唯一。

备选方案：

- 在说明卡中重复放“开始使用”：会把主操作分散到两个位置，增加交互歧义。
- 在说明卡中展示完整治理描述：当前没有必要，会挤占有限空间，也与实际决策不符。

补充布局决策：

- `开始使用` 保持在操作行左侧
- `i` 信息按钮固定在同一操作行右侧

这样可以维持主操作优先级，同时让补充说明入口在视觉上保持对称且不与主动作混淆。

### 3. 第一版采用固定放置规则，而不是自动避让

说明卡先按 starter 分组在三列桌面布局中的位置固定放置：

- 左列：向右展开
- 中列：向下展开
- 右列：向左展开

原因是：

- 当前 starter 区域是稳定的三列结构，固定规则更容易设计、实现和测试。
- 自动避让会引入额外测量逻辑、边界翻转状态和潜在抖动，不适合作为第一版范围。

备选方案：

- 基于视口边界自动翻转：更灵活，但复杂度和测试成本明显更高。
- 全部统一向下展开：虽然简单，但会让中下方内容被更频繁遮挡。

### 4. 说明卡锚定在已展开技能条目内部，通过局部 overflow 放开而非 portal

说明卡将作为已展开技能条目的局部浮层渲染，并依赖该条目与其父卡片的可见 overflow、定位上下文和较高 z-index 完成叠放，而不是额外引入 teleport/portal 层。

选择这一做法的原因是：

- 当前 starter 已展开条目已经允许 `overflow: visible`，与说明卡挂载位置兼容。
- 这块交互仅作用于空会话 starter 区，局部浮层更容易维持样式与状态封装。
- 不需要为第一版新增全局浮层基础设施。

备选方案：

- 使用全局 portal/teleport：对边界更稳，但需要额外的定位、滚动同步和关闭策略，超出当前范围。

### 5. 说明卡正文采用“宽松但有限”的展示上限

说明卡的文本边界固定为：

- 标题单行
- 正文最多 6 行
- 超出截断
- 不滚动

这样既能比 starter 卡本体展示更多摘要内容，又能避免 hover card 变成占用过大的小弹窗。

备选方案：

- 无限展开正文：会显著增加遮挡范围，削弱 tips 的轻量属性。
- 浮层内部滚动：用户阅读成本高，也违背“提示卡而非弹窗”的交互预期。

## Risks / Trade-offs

- [Risk] 固定放置规则依赖当前 starter 三列顺序，未来如果分组顺序或布局结构变化，规则可能不再合适
  → Mitigation：将放置规则显式写入 spec，并把自动避让保留为后续独立演进项。

- [Risk] 局部浮层可能受到局部容器裁剪或 stacking context 影响
  → Mitigation：实现时限制在空会话 starter 区内调节必要的 `position`、`overflow` 和 `z-index`，不做全局样式扩散。

- [Risk] 说明卡正文 6 行上限仍可能截断极长快速开始摘要
  → Mitigation：明确这是有意的可视边界，优先保持工作台密度与阅读稳定性，而不是在 hover card 中容纳无限文本。

## Migration Plan

1. 在空会话 starter 卡片的已展开技能行中加入 `i` 信息按钮与说明卡渲染结构。
2. 复用现有 governed skill name 与 `starterSummaryText` 作为说明卡数据源，并保留“开始使用”为唯一执行入口。
3. 在 starter 三列布局中按固定规则添加说明卡定位样式与文本行数边界。
4. 补充组件与 store 级测试，覆盖触发条件、唯一触发点、说明卡内容来源和正文显示上限。

Rollback strategy:

- 如需回滚，可移除 `i` 入口与说明卡结构，恢复为当前仅展示摘要的 starter 卡片；该 change 不涉及持久化数据迁移。

## Context

当前文件入口分散在两个语义不同的区域：对话框把“给当前消息附加资料”和“上传文件”绑定成一个显式文字按钮，右侧工作区则通过 `新增` 菜单同时承载上传与空白文件创建。这种分布让用户很难形成稳定心智，到底文件是“附加给这次对话”，还是“在工作区里管理和创建”。与此同时，工作区分组仍显示为 `input` / `output`，没有把“模型读取的资料”和“模型生成的结果”转换成用户语言。

这次变更同时跨越 `apps/web` 与 `apps/agent-backend`。前端不仅要重构首页与会话页的对话框文件入口，还要新增拖拽上传状态；后端则要保留既有上传约束与工作区元数据能力，同时删除已经不再符合产品模型的空白文件新建链路。由于这同时涉及交互收敛、API 下线、文案语义收敛与测试调整，因此适合先形成统一设计。

## Goals / Non-Goals

**Goals:**
- 把文件进入系统的主动入口统一到对话框附件流，首页和会话页都使用同一套极简 `+` 触发器。
- 让对话框支持多文件选择与拖拽上传，并把拖拽反馈限制在对话框区域本身。
- 移除工作区右侧的 `新增`/上传/空白文件创建入口，让工作区退回为文件浏览、打开、下载、删除与重命名区域。
- 下线空白文件新建能力及其前后端实现，避免产品继续暴露低价值且分散心智的手工创建流。
- 将工作区分组文案改为 `参考资料` / `生成结果`，明确表达模型读取资料与模型产出结果的角色差异。

**Non-Goals:**
- 不扩展上传格式，仍保持 `TXT / MD / CSV`。
- 不新增工作区批量上传入口、拖拽到整页上传、或全局拖拽遮罩。
- 不引入新的文件分组体系、回收站、版本管理或草稿概念。
- 不改变既有文件打开、保存、下载、删除、重命名的核心语义。

## Decisions

### Decision: 对话框成为唯一主动文件入口，使用极简 `+` 触发器

首页 `HomeStage` 和工作台 `ConversationPane` 都改为使用同一种紧凑 `+` 附件按钮，点击后直接打开受限文件选择器，而不是继续显示 `上传文件` 文字按钮或任何二级菜单。

Rationale:
- 这让“把资料附加到当前对话”成为唯一清晰入口，符合用户当前操作上下文。
- `+` 的信息密度更高，和极简 OpenAI 风格一致，也为发送按钮留出稳定宽度。
- 如果 `+` 再展开菜单，但菜单里只有上传动作，只会增加一步点击而没有收益。

Alternatives considered:
- 保留 `上传文件` 文字按钮：rejected，因为会继续放大它和右侧工作区入口的职责重叠。
- 让 `+` 先弹出菜单：rejected，因为当前没有第二类对话框附件动作，菜单没有价值。

### Decision: 拖拽上传仅作用于对话框外壳，而不是整页

拖拽支持会绑定在首页和会话页的对话框容器上。拖拽进入支持类型文件时，只高亮对话框区域；拖到其他区域不触发全页 drop zone。放下多个文件后，直接复用现有多文件上传链路。

Rationale:
- 对话框是“附加给当前输入”的语义承载位，拖拽范围与操作意图一致。
- 局部 drop zone 能避免整页误触、高亮过重、以及与右侧文件区或历史栏的视觉竞争。
- 这样能保持页面默认极简，只在真正拖拽时出现状态变化。

Alternatives considered:
- 整页拖拽上传：rejected，因为命中范围过大，容易在密集桌面布局里产生误导。
- 只允许拖到 `+` 按钮：rejected，因为命中面积太小，不符合用户预期。

### Decision: 空白文件新建能力整条下线，而不是仅隐藏入口

前端会删除 `create-file` 菜单、API 调用、store 动作与测试；后端会删除空白文件创建路由、默认蓝图与测试，而不是只在 UI 上移除按钮。

Rationale:
- 既然产品已经决定不再支持手工新建空白文件，继续保留死能力只会增加维护成本和调试噪音。
- UI 隐藏但后台保留会让前后端契约失真，也不符合当前仓库“失败显式、不保留静默兼容路径”的治理基线。

Alternatives considered:
- 仅删除工作区按钮，保留 API：rejected，因为这会留下不再被正式支持的孤立 contract。
- 把空白文件新建移回对话框 `+` 菜单：rejected，因为这会重新混淆“附加资料”和“创建文档”。

### Decision: 工作区组名改为 `参考资料` / `生成结果`，内部 group id 可暂时保留

用户可见分组名称改为 `参考资料` 和 `生成结果`。底层 `groupId` 是否继续沿用 `input` / `output` 属于实现细节，可先不作为迁移阻塞条件；但前后端返回给前端的可见 `label` 必须与新命名对齐。

Rationale:
- 这组名称直接对应产品语义：一类是模型读取的资料，一类是模型生成的结果。
- 不强制先改底层枚举名，可以降低这次变更的无效改动面。
- 用户语言和工程标识分离，有利于后续继续演进内部实现而不扰动界面。

Alternatives considered:
- 继续使用 `input` / `output`：rejected，因为过于工程化。
- 改成 `已上传` / `已生成`：rejected，因为更像一次动作结果，不像长期稳定的文件角色。
- 改成 `模型输入` / `模型输出`：rejected，因为虽然准确，但术语感过强，不够产品化。

### Decision: 上传格式帮助不再通过显式辅助控件占据对话框主操作区

对话框不再保留 `i` 提示或任何可见帮助控件。格式限制继续通过文件选择器 `accept`、拖拽前端校验和后端上传校验共同执行，错误通过显式失败反馈暴露。

Rationale:
- 你已经明确要走极简 OpenAI 风格，额外帮助控件会破坏这条视觉原则。
- 格式治理本身已经有前后端双层约束，不需要再用常驻帮助控件占位。
- 去掉辅助控件后，对话框一行结构更稳定，拖拽高亮也更容易收敛。

Alternatives considered:
- 保留原 `i` 提示：rejected，因为视觉上过碎。
- 把格式说明改成 hover tooltip：rejected for now，因为仍然会引入额外 UI 责任点。

## Risks / Trade-offs

- [去掉工作区上传入口后，新用户可能一时找不到文件入口] → 通过对话框 `+` 的显著位置、拖拽上传支持，以及首页与会话页统一入口降低学习成本。
- [拖拽上传会绕过文件选择器 `accept` 限制] → 在前端拖拽处理时重复执行扩展名校验，并保持后端校验为最终真相源。
- [删除空白文件新建 API 可能影响隐藏依赖或测试夹具] → 在实现前先全量搜索 create-file 链路，并同步清理前后端测试与 OpenSpec 要求。
- [继续保留底层 `input` / `output` 标识可能让实现层与 UI 语义暂时分离] → 接受这一层 trade-off，但要求所有用户可见文案和组标签先完成切换。

## Migration Plan

1. 先更新 OpenSpec：移除工作区新增/空白文件创建契约，新增对话框 `+` 与拖拽上传契约，并更新工作区分组命名。
2. 前端重构首页与会话页的对话框附件入口，补充拖拽状态、拖拽校验与多文件上传测试。
3. 前端移除工作区 `新增` 菜单、相关 store/API/types 与空状态文案中的旧描述。
4. 后端删除空白文件创建路由与蓝图实现，保留受限上传、文件打开保存、工作区元数据与分组标签能力。
5. 回归校验工作区列表、上传、拖拽、下载、删除、重命名与已有输出文件显示，确保没有残留 UI 指向已下线能力。

Rollback strategy:
- 如果 `+` 入口或拖拽交互验证失败，可以回滚到文字型上传按钮，但不应恢复工作区侧栏的 `新增`/空白文件创建模型，除非另有明确产品决策。
- 如果删除空白文件创建 API 暂时造成实现阻塞，可以短暂保留后端实现但不得暴露前端入口；不过最终目标仍是彻底移除。

## Open Questions

- 拖拽不支持的文件类型时，错误反馈应优先放在全局状态条、对话框内局部提示，还是两者同时使用。
- 工作区完全空白时，是否需要保留极轻量空态占位文案，还是允许文件区直接呈现纯空列表。

## Context

当前工作区删除能力已经存在，但入口是文件行右侧常驻垃圾桶图标；下载能力则尚未形成正式 contract。我们已经明确首版范围只做“行级更多菜单 + 下载 + 删除”，不把重命名、复制路径、批量管理等后续动作一起纳入。

这意味着本次设计需要同时处理两个层面：

1. 前端交互层：把高风险删除动作从常驻图标降级为二级菜单项，并给文件行补上下载入口。
2. 后端 contract 层：提供一个明确、可鉴权、可按 scope 校验的下载接口，让前端不用复用编辑器打开接口去拼装下载行为。

## Goals / Non-Goals

**Goals:**
- 用行级 `更多` 菜单取代当前常驻删除图标，降低危险动作噪音。
- 首版菜单只提供 `下载` 和 `删除`，保持范围可控。
- 保持文件行单击选中、双击打开的主操作完全不变。
- 为当前 `user + agent` scope 的 upload/output 文件提供显式下载能力。
- 对删除和下载失败都保持显式反馈，不引入静默降级。

**Non-Goals:**
- 不实现重命名、复制路径、另存为副本、批量选择等扩展文件动作。
- 不重做顶部 `新增` 菜单，也不顺带抽象整个菜单系统。
- 不修改工作区文件保存、打开、删除的既有业务语义。
- 不引入新的文件版本管理、回收站、软删除或恢复机制。

## Decisions

### Decision: 文件行使用专用行级更多菜单，而不是继续暴露常驻删除图标

工作区文件行将拆分为两个明确区域：

```text
[主文件行点击区] [更多]
               ├ 下载
               └ 删除
```

菜单 trigger 负责承载次级操作，主文件行继续承担选中和打开。点击菜单 trigger 或菜单项时，不得触发文件行选中或打开。

Rationale:
- 删除属于高风险动作，不应长期作为常驻显性图标占据每一行。
- 下载和删除都属于文件次级动作，放入同一个二级菜单更符合层级。
- 这个结构为后续扩展动作保留空间，但首版又不会把菜单做得过重。

Alternatives considered:
- 保留垃圾桶图标并额外新增下载图标： rejected，因为会让行内操作区继续膨胀，风险动作仍然过度外露。
- 只把删除改成文字按钮： rejected，因为它解决了图标语义问题，但没有解决下载入口承载位问题。
- 直接抽象通用菜单框架： rejected for now，因为首版目标是快速收敛，不扩大组件重构面。

### Decision: 下载使用独立后端 endpoint，而不是复用打开文件接口

后端将新增 `GET /agent/api/files/:fileKey/download?agentId=<id>` 路由。该路由复用当前文件 API 的鉴权、scope 解析与 `fileKey` 解析逻辑，但返回附件下载响应，而不是编辑器打开 payload。

后端响应将以文件存储内容为权威来源，并返回明确的附件元数据：
- 文件内容来自当前 tracked workspace entry 对应的真实磁盘文件
- 响应头必须包含权威下载文件名，前端不需要猜测或拼装文件名

Rationale:
- 打开文件接口面向编辑器，下载接口面向浏览器附件语义，两者职责不同。
- 独立 endpoint 可以让前端直接触发浏览器下载，而不依赖编辑器模式转换或文本内容拼接。
- 这样可以避免未来 output 文件格式扩展时，下载行为被编辑器展示逻辑污染。

Alternatives considered:
- 前端先调用打开接口再用文本内容生成下载： rejected，因为这会把“编辑器视图数据”和“文件下载 contract”混为一体。
- 让前端直接拼接工作区路径下载静态文件： rejected，因为需要绕过当前 `user + agent` scope 鉴权与 fileKey 映射。

### Decision: 首版菜单严格限制为 `下载` 在前、`删除` 在后

菜单首版仅提供两个动作，并固定顺序为：

1. `下载`
2. `删除`

其中 `删除` 保持危险态样式，并继续沿用现有确认提示与删除后状态收敛逻辑。

Rationale:
- 用户已经明确希望先按简单方案推进。
- 菜单项越少，越不容易削弱文件行的主操作优先级。
- `下载 + 删除` 已经覆盖了最直接的次级需求，不需要为了“以后可能会有”先堆更多动作。

Alternatives considered:
- 首版同时加入重命名： rejected，因为会把范围从“菜单承载”扩大到“文件生命周期编辑”。
- 菜单里加入 `打开`： rejected，因为与主文件行交互重复。

## Risks / Trade-offs

- [行级菜单浮层在滚动侧栏里被裁切] → Mitigation: 菜单定位与容器样式需要按侧栏滚动结构校验，测试覆盖顶部/底部文件行。
- [菜单 trigger 与文件行点击事件串扰] → Mitigation: 明确阻断 trigger 和菜单项事件冒泡，并补充组件测试。
- [下载接口遗漏 filename header，导致前端只能猜文件名] → Mitigation: 将权威文件名写入后端 contract，前端按 header 驱动下载。
- [首版菜单过轻，后续还要扩展] → Trade-off accepted: 当前优先解决交互层级问题，不提前做过度抽象。

## Migration Plan

1. 在 agent-backend 文件路由增加下载 endpoint，并补充作用域、缺失文件、显式失败测试。
2. 在 web 端增加工作区文件下载 API 调用与浏览器下载触发逻辑。
3. 将 WorkspaceSidebar 的常驻删除图标替换为行级更多菜单，菜单内放入 `下载` 与 `删除`。
4. 保持现有删除确认与状态收敛逻辑，仅更换触发入口。
5. 更新 Sidebar、Shell、store 和后端路由测试，覆盖菜单交互与下载/删除行为。

Rollback strategy:

- 移除下载 endpoint 与前端下载动作。
- 将文件行菜单恢复为当前独立删除图标入口。
- 由于本次不引入数据迁移或存储结构变化，回滚不涉及持久化数据修复。

## Open Questions

- None. 当前范围已经明确收敛为首版 `下载 + 删除`。

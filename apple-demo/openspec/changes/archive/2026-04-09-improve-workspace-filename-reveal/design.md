## Context

当前工作区侧栏在 `apps/web` 中以树状文件列表呈现 `input` / `working` 内容，行内文件名使用单行省略来维持高密度布局。这个策略本身符合工作台侧栏的扫描场景，但长文件名会带来三个交互缺口：

- 用户上传文件后，无法在侧栏稳定确认完整文件名；
- 文件扩展名在窄宽度下缺少明确的可辨识性保障；
- “复制文件名”没有与现有文件操作形成统一入口。

现有组件边界已经具备承载这次改动的基础：

- `WorkspaceSidebar.vue` 负责工作区树与行级文件操作；
- `WorkspaceFileActionMenu.vue` 已经承载重命名 / 下载 / 删除菜单；
- `WorkspaceEditorPane.vue` 已经有 context bar、toolbar、status bar，可承载“当前选中文件”的稳定信息。

本次改动只涉及前端工作区体验，不需要调整顶层目录结构，也不引入新依赖或后端接口。

## Goals / Non-Goals

**Goals:**

- 保持工作区侧栏紧凑的单行树状布局，不因长文件名改成默认多行。
- 让被截断的文件名在 hover / focus 场景下可以查看完整名称。
- 为当前选中的文件提供稳定的完整文件名展示位，避免完整名称只能依赖悬浮态。
- 将“复制文件名”并入统一的文件动作层级，与重命名 / 下载 / 删除保持一致。
- 在侧栏窄宽度场景下优先保留扩展名的可辨识性。

**Non-Goals:**

- 不引入新的独立详情面板或重构整个工作区布局。
- 不处理移动端专用长按交互；本次以桌面工作台场景为主。
- 不扩展为“复制路径”“复制链接”等更多复制语义。
- 不针对大批量相似前缀文件名做差异高亮或复杂智能对比。

## Decisions

### 1. 侧栏继续保持单行紧凑布局，但改为“压缩 + 揭示”而不是“压缩 + 丢失”

侧栏的首要职责仍然是快速扫描，因此不采用多行换行或自动增高行高。长文件名将继续被压缩在单行中，但压缩后的信息不能丢失。

备选方案：

- 改成两行或自动换行：完整性更高，但会显著破坏树状列表密度与节奏，放弃。
- 保持现状只做尾部省略：实现最简单，但无法解决完整名称确认问题，放弃。

### 2. 侧栏长文件名优先保留扩展名可辨识性

由于用户明确认为扩展名重要，长文件名展示不应只做纯尾部省略，而应采用保留文件尾部的截断策略，优先确保扩展名仍可见。具体实现可以是中间截断，或其他等价的“保尾部”策略，但输出结果必须满足扩展名可辨认。

备选方案：

- 纯尾部省略：会丢失扩展名与版本尾缀，不符合场景优先级，放弃。
- 完整显示文件名：会挤压树状布局，放弃。

### 3. 完整文件名采用双通道揭示：悬浮揭示 + 选中态稳定展示

完整文件名不只依赖 tooltip。具体分工如下：

- 侧栏 hover / keyboard focus：用于临时查看完整名称；
- 选中后的编辑区上下文栏：用于稳定查看当前文件的完整名称。

这样可以把“扫描”与“确认”分开，既不让侧栏承担完整信息展示职责，也不把 tooltip 变成唯一信息来源。

备选方案：

- 只做 tooltip：复制和稳定确认都不友好，放弃。
- 只做选中态展示：未选中文件仍难快速确认，放弃。

### 4. “复制文件名”进入统一文件动作菜单，不作为悬浮 tooltip 的职责

复制动作应与重命名 / 下载 / 删除同层级，避免新增一个高干扰的独立复制按钮。选中态的职责是稳定显示完整名称，复制动作则通过文件动作菜单触发。

为提升可发现性，当前选中文件的稳定信息区域应直接关联同组文件动作，而不是要求用户回到侧栏树中寻找对应行。

备选方案：

- 在完整文件名旁直接放独立复制按钮：可发现性高，但会让信息栏过于工具化，暂不采用。
- 只在侧栏行菜单中增加复制：虽然复用性高，但会削弱“当前选中文件”场景的便捷性，放弃。

### 5. 复制能力使用浏览器 Clipboard API，失败时显式反馈

复制文件名由前端直接调用浏览器 Clipboard API 完成，不新增后端接口。若复制失败，界面必须给出明确错误反馈，而不是静默失败或伪造成功状态。

备选方案：

- 通过隐藏输入框和 `execCommand('copy')` 做兼容降级：会引入额外兼容路径，且收益有限，本次不作为主设计目标。
- 后端参与复制：无必要，放弃。

## Risks / Trade-offs

- [中间截断实现细节不当] → 统一封装文件名压缩逻辑，并通过测试覆盖无扩展名、隐藏文件、多后缀文件。
- [tooltip 只对鼠标友好] → 明确要求 keyboard focus 同样可揭示完整名称。
- [选中态新增信息栏后挤压现有 toolbar] → 将完整文件名放在已有上下文栏中，与视图切换和保存操作分层，避免把所有元素塞进同一行。
- [复制动作存在但用户不易发现] → 在当前选中文件的稳定信息区域直接提供同组文件动作入口，而不只保留侧栏行菜单。
- [Clipboard API 权限或环境限制导致复制失败] → 复制失败时提供显式错误提示，不做静默 fallback。

## Migration Plan

1. 在前端组件层新增文件名压缩 / 完整名称揭示规则，并保持现有数据模型不变。
2. 扩展文件动作菜单，加入“复制文件名”动作，并在当前选中文件的稳定上下文中暴露该动作入口。
3. 更新相关组件与 store 测试，覆盖长文件名、扩展名可见性、复制反馈和选中态展示。
4. 由于改动仅限前端交互，无需数据迁移；如出现回归，可回退对应前端组件改动。

## Open Questions

- 选中态完整文件名最终放在 context bar 的独立标题行，还是合并进现有 toolbar 主区；实现前需要按现有布局做一次视觉权衡。
- 当前复制成功后的反馈是沿用全局轻量状态文案，还是在动作菜单附近提供更局部的确认提示；可在实现时按现有 workbench 反馈模式统一。

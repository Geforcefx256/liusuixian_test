## Context

`Skill 管理` 页面已经完成两轮收口：先解决 detail pane 滚动责任，再把字体语义向工作台基线对齐。但截图反馈表明，当前页面仍存在明显的视觉失衡：字体已经缩小，容器、圆角、控件和留白却仍保留更大的内容页尺度，导致左栏 skill 项尤其显得“框大于内容”。

这次变更不涉及治理模型或运行时能力，而是一次面向 workbench surface 的视觉密度调整。重点是把页面从“松散卡片页”收口成“高密度治理台”，同时保留现有信息架构和交互行为。

## Goals / Non-Goals

**Goals:**
- 将左栏 skill 列表项改为更紧凑的治理列表表达，采用 A2 选中态。
- 收紧右栏 section、metadata strip、表单控件和 starter 预览的壳体尺度，使其与当前字体层级匹配。
- 让整页形成一致的形状语言和密度节奏，而不是只局部修左栏。
- 保持桌面态双栏独立滚动和窄屏回退行为不变。

**Non-Goals:**
- 不修改业务字段、保存 payload、后端治理逻辑或运行时元数据。
- 不引入新的全局设计 token，也不重做整个 workbench 的设计系统。
- 不改变 `Skill 管理` 的信息架构与字段语义。

## Decisions

### Decision: 左栏列表采用 A2 选中态

选中 skill 项使用左侧 accent rail、轻边框和轻底色变化，不再依赖更厚的阴影卡片效果。

Rationale:
- 更适合“治理目录”而不是“内容推荐卡”。
- 在压缩列表项高度后，仍能快速识别当前选中项。

Alternatives considered:
- 继续使用厚阴影卡片：视觉重量过高，和高密度目标冲突。
- 只保留边框高亮：识别度不足，连续列表里不够稳定。

### Decision: 整页密度一起收口，而不是只改左栏

虽然红框问题最先暴露在左栏，但根因是整页字体和容器尺度失配，因此需要同步调整 hero、主卡、section、控件和 pill。

Rationale:
- 只改左栏会让右栏仍然显得壳体过厚，页面气质不统一。
- 顶部、左右主卡和 section 使用相近的壳体语言，必须一起收口才会稳定。

Alternatives considered:
- 只做左栏列表：实现更小，但不能解决“字小框大”的整页问题。

### Decision: 仅做页面局部密度规则，不改全局 token

本次在 `AdminSkillManagement.vue` 内部调整圆角、padding、gap、控件高度和摘要截断；如有必要只提取最小量的局部变量，不修改全局工作台 token。

Rationale:
- 问题集中在这页，不代表整个 workbench 都需要缩。
- 风险更低，回归面更小。

Alternatives considered:
- 修改全局 card/control token：影响面过大，和当前问题不匹配。

## Risks / Trade-offs

- [页面收得过紧导致可读性下降] → 保持标题/摘要/meta 的层级清晰，只收壳体和空白，不继续压字体。
- [只靠样式调整引发响应式回归] → 保留现有桌面/移动滚动链，并补源码级样式回归测试。
- [A2 选中态过轻导致选中项不明显] → 使用左侧强调条叠加轻边框和浅底色，而不是只改边框。

## Migration Plan

1. 收紧左栏列表项和 A2 选中态。
2. 同步收紧顶部、右栏 section、控件和 starter 预览卡。
3. 保持现有滚动布局不变，补前端回归测试验证新的密度规则。

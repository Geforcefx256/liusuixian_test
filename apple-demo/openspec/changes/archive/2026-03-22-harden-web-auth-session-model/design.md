## Context

`build-vue-agent-workbench-lite` 已经完成了 Vue 工作台与最小认证闭环，当前系统已经可以基于 Cookie session 完成本地登录、OAuth 回调、当前用户加载和退出登录。但这套能力仍偏向 phase-1 的单机可跑通形态，几个关键边界尚未稳定：

- auth mode 目前更像前端提示，而不是后端强语义配置。
- OAuth `state` 依赖进程内临时内存，无法稳定覆盖服务重启、长等待跳转和多实例部署。
- session 生命周期只有“自然过期”和“显式登出”两类弱约束，改密和用户禁用后的行为不明确。
- OAuth 首次登录时的身份绑定规则不完整，后续容易出现账号冲突、重复用户和默认角色漂移。

这次设计要解决的是认证模型收敛问题，而不是继续扩展更多工作台业务能力。

## Goals / Non-Goals

**Goals:**
- 将 auth mode 从“偏好提示”升级为后端强语义配置。
- 将 OAuth 登录启动与 `state` 校验收敛成可持久化、可消费的一次性事务。
- 为 session 建立统一的失效规则，覆盖登出、过期、改密和用户禁用等场景。
- 明确 OAuth 身份与本地用户之间的绑定、建号与冲突处理原则。
- 为前端提供稳定的 auth 契约，使登录入口和错误恢复与后端语义一致。

**Non-Goals:**
- 不在本次变更中交付完整管理员用户管理后台。
- 不在本次变更中重构完整 RBAC 或细粒度权限体系。
- 不在本次变更中引入独立 IAM 中间层或新的 BFF。
- 不在本次变更中扩展工作台业务功能或文件处理能力。

## Decisions

### 1. 认证模式采用强语义配置，而不是仅由前端决定入口展示

系统将认证模式明确分为三类：

- `local_only`
- `oauth_preferred`
- `oauth_only`

其中：

- `local_only` 允许本地登录，不暴露 OAuth 入口。
- `oauth_preferred` 同时允许本地登录和 OAuth，但前端默认优先引导 OAuth。
- `oauth_only` 只允许 OAuth，后端必须拒绝本地密码登录。

原因：

- 当前 `preferred: oauth` 但本地登录仍可用，会让产品语义和安全边界不一致。
- 后续管理员后台和运维策略都需要以明确模式为准，而不是依赖前端显示逻辑。

备选方案：

- 继续保持“preferred + enabled”的弱语义。
  - 放弃原因：无法表达真正的后端能力边界，也无法消除隐藏旁路。

### 2. OAuth 登录事务必须显式创建并持久化，`/auth/mode` 不再生成 `state`

目标模型如下：

```text
Browser
  │
  ├─ GET /auth/mode
  │      └─ 只返回模式与可用登录方式
  │
  ├─ GET /auth/login-url
  │      └─ 服务端创建一次性 OAuth login transaction
  │         - state
  │         - created_at / expires_at
  │         - optional return_to
  │
  ├─ redirect to provider
  │
  └─ callback with code + state
         └─ 服务端校验并消费 transaction
            └─ 建立本地 session
```

设计约束：

- `/auth/mode` 只做能力发现，不承担事务生成副作用。
- OAuth login transaction 必须具备 TTL、单次消费和显式过期语义。
- transaction 存储必须脱离进程内 `Map`，至少能覆盖服务重启；设计上兼容多实例。
- 前端使用过期 `loginUrl` 时，系统必须能够给出可恢复行为，而不是落入随机失败。

原因：

- `mode` 是配置发现接口，不应顺带创建有状态登录事务。
- 这能避免服务重启、多标签页和长时间停留后使用旧 `state` 的脆弱路径。

备选方案：

- 继续用进程内内存存储 `state`。
  - 放弃原因：仅适合单实例开发，不适合作为正式模型。

### 3. Session 采用服务端治理模型，失效规则由用户状态和安全事件驱动

系统将 session 视为服务端可治理对象，而不是只依赖浏览器 cookie 是否仍在。

推荐规则：

- 登录成功时创建新 session。
- 登出当前设备时，仅撤销当前 session。
- 用户被禁用后，所有已有 session 都必须失效。
- 本地密码修改后，该用户已有 session 必须全部失效。
- session 过期由服务端判定，前端只消费结果。
- 是否允许同一用户并发持有多个 session，本次不做单活限制，但要求行为明确可测试。

原因：

- “改密但旧 cookie 仍可继续使用”会直接削弱改密的安全意义。
- 后续管理员后台会需要解释“为什么这个登录态失效”，因此底层必须先定义可解释规则。

备选方案：

- 只保留自然过期和显式登出。
  - 放弃原因：无法支撑最基本的账号治理场景。

### 4. OAuth 身份绑定以外部身份为稳定键，不按邮箱或登录名静默合并

身份模型拆成三层：

```text
External Identity
(provider_code + external_user_uuid)
          │
          ▼
Identity Binding
          │
          ▼
Local User
```

核心原则：

- `provider_code + external_user_uuid` 是 OAuth 身份的唯一稳定键。
- `loginName` 和 `email` 仅作为辅助属性，不作为隐式合并依据。
- 首次 OAuth 登录时是否自动建号必须是明确策略，而不是代码副作用。
- 若 provider profile 与现有本地用户产生账号冲突，系统必须返回显式冲突结果。
- 默认角色赋予需要有明确策略，避免随着实现路径漂移。

原因：

- 邮箱和登录名都可能变化，不适合作为强绑定主键。
- 当前直接按 provider 返回字段建本地账号，未来在账号冲突和多 IdP 场景下会迅速变脆。

备选方案：

- 继续按 `loginName` 自动建号并在冲突时依赖数据库约束失败。
  - 放弃原因：冲突路径不可解释，也不利于后续用户治理。

### 5. 开发便利性与生产安全性显式分层

默认本地 `admin` 账号对开发环境有价值，但不应自然外溢到正式认证模式。

设计约束：

- 开发辅助账号策略需要显式与环境或模式绑定。
- `oauth_only` 模式下不应保留可通过密码登录的隐式超级管理员入口。
- 若未来需要 break-glass 账户，应作为单独运维策略设计，而不是沿用开发默认值。

原因：

- 这类“默认方便”的入口一旦进入正式环境，通常会成为长期旁路。

## Risks / Trade-offs

- [认证模式语义收紧后，现有前端和部署脚本需要同步调整] → 通过兼容映射与清晰配置文档降低切换成本。
- [一次把身份绑定做太深，容易膨胀成完整用户系统重构] → 本次只定义绑定规则和冲突策略，不扩展管理员后台流程。
- [将 OAuth transaction 和 session 治理落在现有 SQLite 上，后续仍可能继续演进] → 先保证单实例重启恢复和设计兼容多实例，再为未来存储升级留扩展位。
- [自动建号策略若过于保守，会增加首次接入成本] → 允许受控自动建号，但要求冲突路径显式化、可测试。

## Migration Plan

1. 定义新的 auth mode 语义以及与现有配置的兼容映射。
2. 将 OAuth `state` 签发从 `/auth/mode` 剥离到显式登录启动接口。
3. 引入可持久化的 OAuth login transaction 存储与过期/消费语义。
4. 收敛 session 失效规则，覆盖改密、禁用、登出与过期场景。
5. 调整前端 auth store，使模式发现、登录启动和错误恢复符合新契约。
6. 定义首次 OAuth 登录与现有账号冲突时的返回语义。
7. 补充测试，覆盖 auth mode、OAuth callback、session 失效和身份冲突路径。

回滚策略：

- 保留现有本地登录和最小 session 闭环作为回退基线。
- 若 OAuth transaction 持久化方案引入兼容问题，可临时回退到仅支持本地登录的开发模式，但不回退规范结论。

## Open Questions

- 本次是否需要同时暴露“登出其他设备”能力，还是只先定义底层 session 失效规则。
- 首次 OAuth 登录是否允许自动建号进入，还是需要管理员预置用户后才能进入。
- 开发环境默认本地 `admin` 的保留方式，是否需要单独配置开关。
- OAuth login transaction 是否直接复用现有 SQLite，还是抽象独立 auth 存储接口。

## Context

当前 `WorkspaceEditorPane` 的顶部由三个连续区块组成：标签行、当前文件区块、工具栏。标签行已经表达当前激活文件，当前文件区块又再次展示“当前文件 + 文件名”，导致顶部在视觉和语义上都存在重复。该重复区块同时承载文件级操作菜单，使得删除冗余信息时需要一并明确文件操作入口的新归属。

这次变更仅涉及前端工作台编辑器头部结构，不修改后端协议、不引入新依赖，也不改变工作台三栏总体布局。

## Goals / Non-Goals

**Goals:**
- 将编辑器顶部收敛为稳定的两层结构：文件标签行与工具栏行。
- 去除重复文件名表达，确保当前文件名只由标签行承担。
- 保留文件级操作入口，并将其放置到不依赖重复标题的位置。
- 保持桌面与窄屏下的信息职责一致，避免因换行重新引入重复标题。

**Non-Goals:**
- 不重做工作台整体视觉风格。
- 不调整编辑器正文、状态栏或文件树的行为。
- 不改变文件保存、搜索、MML 配置、更多菜单的业务逻辑。
- 不引入 breadcrumb、完整路径展示或新的文件元信息模块。

## Decisions

### 决策 1：删除 `selected-file` 区块，保留标签行作为唯一文件名入口

当前文件名已经在标签行中明确表达，再额外渲染“当前文件 + 文件名”不会提供新增信息。删除该区块后，顶部层次将变为：

- 第一层：文件标签导航
- 第二层：视图切换、MML 状态、编辑动作

备选方案：
- 保留当前文件区块但只显示更弱的标题。否决，因为仍会占用空间且缺乏新信息。
- 将当前文件区块改成元信息行。此次不采用，因为用户当前只确认了去重，不需要扩大范围到新信息设计。

### 决策 2：文件级操作入口迁移到标签行右侧

文件级操作是有效能力，不应跟随冗余标题一起消失。操作入口应直接依附文件导航层，而不是依附重复文件标题层。这样用户可以在“文件上下文”里找到文件动作，在“工具栏”里找到编辑动作。

备选方案：
- 将文件级操作合并进工具栏 `更多`。否决，因为会混淆文件级操作与编辑器级操作。
- 将文件级操作隐藏到活动标签内。否决，因为会提高 discoverability 成本，并增加标签交互复杂度。

### 决策 3：窄屏下保持两层职责，不回退到重复标题模式

窄屏时允许标签行与工具栏换行或纵向堆叠，但不新增单独的“当前文件”标题行。响应式调整应只影响排布，不改变信息职责。

备选方案：
- 在窄屏单独恢复当前文件标题。否决，因为会造成跨断点语义不一致。

## Risks / Trade-offs

- [文件操作入口位置变化可能降低短期可发现性] → 保持入口仍在头部右侧，并延续现有三点菜单样式与语义。
- [移除区块后顶部留白和对齐关系会变化] → 同步清理相关样式，确保标签行与工具栏之间的间距仍然稳定。
- [现有测试依赖 `selected-file` 结构] → 更新组件测试，改为断言“文件名不重复渲染”与“文件操作入口仍可用”。
- [窄屏布局可能因换行出现拥挤] → 限定为排布重组，不增加新控件，优先保证操作分组和可点击面积。

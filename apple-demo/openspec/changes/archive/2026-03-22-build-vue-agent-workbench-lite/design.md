## Context

当前仓库的后端已经迁移到 `apps/agent-backend`，并暴露了智能体目录、runtime bootstrap、会话、消息历史、流式执行和文件上传等接口。与此同时，仓库根目录的 `index-v10.html` 已经提供了新的产品形态：它不是旧工作台里的 AI 侧栏，而是一个以智能助手为中心的独立工作台。

现状的问题是：

- `apps/web` 仍为空壳，没有正式前端工程。
- `index-v10.html` 只是静态原型，内部数据全部是前端硬编码，没有真实接口接入。
- `ref_code/apps/web` 有较完整的 Vue 前端和认证/智能体接入经验，但它带有旧工作台和历史桥接逻辑，不适合作为新产品壳直接整体平移。

这次设计需要在三者之间做收敛：

- 以 `index-v10.html` 提供新的页面形态和交互布局。
- 以 `ref_code/apps/web` 提供 Vue、Pinia、认证接入和 agent API 接入参考。
- 以 `apps/agent-backend` 作为新的真实后端依赖。

## Goals / Non-Goals

**Goals:**
- 在 `apps/web` 下建立一个基于 Vue 的正式前端工程，而不是继续维护纯静态 HTML 原型。
- 将 `index-v10.html` 的首页、会话栏、聊天区和右侧工作区上下文区落为真实可用页面。
- 接入 Cookie 会话认证，让未登录用户无法进入工作台，并在已登录状态下展示当前用户和登出入口。
- 接入 agent 列表、runtime bootstrap、session、message history、run stream 和文件上传接口，形成最小可用闭环。
- 将首期工作区定义为“上下文与文件列表区域”，避免误导性地引入未完成的编辑器/预览体验。

**Non-Goals:**
- 不在本次变更中实现文件内容预览、表格编辑、文本编辑、sheet 汇总或下载工作台。
- 不在本次变更中迁移旧前端的完整配置管理、模板管理和 Excel 桥接逻辑。
- 不在本次变更中实现管理员用户管理后台、角色管理或使用统计面板。
- 不在本次变更中重构 `apps/agent-backend` 的接口语义，只消费已有接口。

## Decisions

### 1. 新前端以 Vue 3 + Vite + Pinia 为基础，而不是继续扩展单文件 HTML

原因：
- `ref_code/apps/web` 已经证明了 Vue 3 + Pinia 的使用方式，迁移心智成本最低。
- 当前原型已经超过简单静态页的复杂度，后续需要认证、会话、流式消息和上传状态，必须引入正式状态管理。
- 使用 Vite 可以直接延续旧前端的开发模式，降低新应用初始化成本。

备选方案：
- 继续以原生 HTML/JS 演进 `index-v10.html`。
  - 放弃原因：状态复杂度已经超过这种模式的舒适区，后续维护成本会快速上升。
- 直接平移 `ref_code/apps/web` 整个应用。
  - 放弃原因：旧应用与历史工作台桥接耦合严重，不符合新的产品壳。

### 2. 新工作台采用单一 Vue 应用壳，登录页与工作台作为同一前端工程内的不同页面状态

原因：
- 新产品更像独立工作台，不需要沿用旧系统多个 HTML 入口的历史包袱。
- 统一在一个应用中管理登录态、当前用户、路由守卫和工作台初始化更清晰。
- 这能让 `index-v10.html` 的页面切换逻辑自然演化为组件状态或路由状态，而不是多个静态入口。

备选方案：
- 继续使用旧系统那种 `login.html` + `index.html` 的多入口模式。
  - 放弃原因：虽然能工作，但会把旧工程边界直接带入新产品，不利于后续扩展。

### 3. `index-v10.html` 的中间工作区首期收敛为“聊天 + 上下文”，不实现编辑器/预览

原因：
- 用户已经明确首期重点是智能助手、工作区域和整体对话流，不做文件预览。
- 原型中的表格/文本区如果原样保留，会让用户预期“可以查看和编辑文件内容”，与首期范围冲突。
- 更合理的首期做法是把右侧与中间非聊天区域定义为任务上下文、上传文件列表、当前结果摘要和后续动作入口。

备选方案：
- 保留表格视图和文本视图，但先用静态占位。
  - 放弃原因：容易制造错误产品预期，也会拉高首期实现复杂度。

### 4. 认证能力只做最小闭环，完整用户管理延后

原因：
- 首期目标是验证“已登录用户能否顺畅使用智能体工作台”，而不是交付完整后台。
- `ref_code` 的用户管理后端范围较大，包括管理员列表、角色、统计与会话，不适合首期全部迁入。
- 只实现认证模式发现、登录、当前用户、登出，就足以支撑工作台闭环。

备选方案：
- 首期同时交付管理员用户管理界面。
  - 放弃原因：会分散主线目标，且不影响智能体工作台的最小可用性。

### 5. 智能体前端优先围绕现有 backend 接口建模，不额外设计新的 BFF

原因：
- `apps/agent-backend` 已经具备会话、消息历史、流式执行、bootstrap 和上传入口。
- 首期的主要风险在前端收敛，而不是后端接口不足。
- 继续使用同源代理 + Cookie 会话，也与当前 backend 设计一致。

备选方案：
- 先创建新的前端聚合层或 BFF。
  - 放弃原因：会把一个前端收敛任务扩展成接口架构重构。

## Risks / Trade-offs

- [`index-v10.html` 的静态结构与真实组件树不完全对齐] → 以信息架构和布局为准迁移，不强求 DOM 一比一复刻。
- [旧前端的 agent store 与新布局耦合度不足或过高] → 复用 API 客户端与交互模型，避免直接搬运整套 store。
- [认证模式同时支持 OAuth 和本地账号时，首次体验存在跳转复杂度] → 复用旧系统的 auth mode 探测逻辑，并优先保持行为一致。
- [首期去掉文件预览后，工作区价值感下降] → 明确把工作区设计成“任务上下文 + 上传文件 + 结果摘要”而不是空编辑器。
- [后续再引入模板库或预览时需要改布局] → 首期保留布局扩展位，但不在本次实现中承诺完整能力。

## Migration Plan

1. 在 `apps/web` 初始化新的 Vue 前端工程，确定构建方式和运行入口。
2. 将 `index-v10.html` 拆解为 Vue 组件和页面状态，保留首页、会话、聊天区和右侧工作区骨架。
3. 接入 `/web/api/auth/*`，建立登录流程、登录态校验、当前用户和登出能力。
4. 接入 `/agent/api/*`，打通 agent bootstrap、session 列表、session 创建、消息历史和流式执行。
5. 接入 `/agent/api/files/upload`，在工作区上下文区展示上传结果和文件列表。
6. 移除或延后原型中的文件预览/编辑器逻辑，改成首期上下文展示。
7. 完成联调和基础测试。

回滚策略：
- 保留现有 `index-v10.html` 原型不动，作为视觉和交互参考。
- 若新前端实现受阻，不影响已迁移的 `apps/agent-backend` 继续作为后端基线运行。

## Open Questions

- 新前端是否会在首期引入 Vue Router，还是在单页内部用应用状态切换登录页与工作台。
- 右侧“模板库”标签首期是完全隐藏、静态占位，还是保留入口但显示“即将开放”。
- 首期工作区上下文区是否需要展示 agent 返回的结构化结果摘要，还是只展示上传文件和任务状态。

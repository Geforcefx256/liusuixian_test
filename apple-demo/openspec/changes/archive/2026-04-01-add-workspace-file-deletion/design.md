## Context

当前工作区文件流只覆盖上传、新建、打开和保存。后端 `files` 路由没有显式删除接口，`fileStore` 只有 TTL 清理时才会实际删除文件与索引；前端 `agentApi`、`workbenchStore` 和 `WorkspaceSidebar` 也都没有删除动作或行级入口。与此同时，工作区文件是按 `user + agent` 复用的，用户切换会话后仍会看到同一批上传文件和 `outputs/` 文件，因此缺少删除能力会持续累积过期输入、手工新建文件和运行时产物。

这个变更跨越 `apps/agent-backend` 和 `apps/web` 两个应用，并且会改变工作区 contract 与前端交互模型，因此需要在实现前固定删除流、确认提示和状态收敛规则。

## Goals / Non-Goals

**Goals:**
- 为当前 `user + agent` 工作区提供显式文件删除能力，覆盖上传文件、手工新建文件和 `outputs/` 下的运行时产物。
- 在前端工作区 Sidebar 的每个文件行提供独立删除图标，不破坏单击选中、双击打开的既有交互。
- 在删除前向用户明确暴露高风险提示，包括不可恢复、可能影响当前会话/后续执行，以及未保存修改将丢失。
- 删除成功后刷新工作区 metadata，并自动收敛侧栏选中态、已打开标签和编辑器状态。
- 保持 debug-first 原则：运行中的会话若后续访问已删除文件而失败，错误必须直接暴露，不引入静默 fallback。

**Non-Goals:**
- 不新增回收站、软删除、撤销删除或版本恢复能力。
- 不为工作区文件引入新的分类体系，例如额外区分“用户新建输出”和“运行时产物输出”。
- 不改变当前 `user + agent` 作用域的工作区复用模型。
- 不在本次变更中新增重命名、移动、下载或批量删除文件能力。

## Decisions

### Decision: 后端通过统一文件删除接口删除真实文件与索引

后端将在现有 `/files/:fileKey` 路由组下新增 `DELETE` 接口，并按当前登录用户与 `agentId` 解析出的 scope 删除目标工作区文件。删除流程由 `fileStore` 提供公开方法统一处理：解析 entry、校验 scope、删除磁盘文件、移除内存索引、持久化更新后的 `file-map.json`。

Rationale:
- 删除是现有文件打开/保存 contract 的自然补充，放在同一路由组能保持接口一致性。
- 由 `fileStore` 统一处理磁盘与索引删除，可以避免在 route 层散落文件系统逻辑。
- scope 校验必须保持在删除流中，避免仅凭 `fileKey` 做跨用户或跨 agent 删除。

Alternatives considered:
- 仅在 route 层直接 `unlink` 文件：会让 route 层承担索引与持久化细节，职责不清。
- 只从 workspace metadata 中移除，不删除磁盘文件：会留下实际文件残留，不满足用户“删除文件”的预期。

### Decision: 前端删除入口放在 Sidebar 行级独立图标，不复用整行点击区

Sidebar 当前每个文件节点是整行按钮。为避免“删除按钮触发行打开/选中”的冲突，文件行将拆为“主点击区 + 删除图标”两块：主点击区负责选中与双击打开，删除图标只负责触发删除确认，且必须阻断事件冒泡。

Rationale:
- 行级删除是用户已确认的入口位置，最符合工作区清理的心智模型。
- 独立删除按钮能保留既有选中/打开交互，不需要改动用户对主文件行的使用方式。
- 相比三点菜单，单一危险动作用纯图标更直接，交互成本更低。

Alternatives considered:
- 在编辑器工具栏里删除当前文件：只能删除已打开文件，不覆盖用户在侧栏快速清理文件的主要场景。
- 使用三点菜单承载删除：为单一动作增加额外点击层级，且需要更多焦点与弹层管理。

### Decision: 风险提示只确认，不阻止

前端删除前统一弹出确认框，明确提示删除不可恢复、可能影响当前会话或后续执行；如果目标文件存在未保存修改，还要额外强调未保存内容会丢失。即使文件属于 `outputs/`、是当前活动文件，或当前会话正在运行，也允许用户继续删除。

Rationale:
- 这与用户确认的产品策略一致：要提醒风险，但不能因高风险而阻断用户。
- 这也符合仓库的 debug-first 规则，不能通过前端保护逻辑掩盖真实依赖问题。

Alternatives considered:
- 运行中禁止删除当前文件：会引入新的保护边界，与已确认需求冲突。
- 自动保存脏文件后再删除：会隐式改变文件状态，掩盖用户删除未保存内容的风险。

### Decision: 删除后通过刷新 workspace metadata 驱动状态收敛

前端在删除成功后重新拉取工作区 metadata，再复用现有 `reconcileWorkspaceState` 收敛本地状态，包括去除失效的 `selectedWorkspaceFileId`、`openWorkspaceFileIds`、`activeWorkspaceFileId` 与编辑器缓存。如果删除的是当前活动文件，编辑器切换到剩余打开文件；如果没有剩余文件，则退出 workspace-expanded 状态。

Rationale:
- 当前工作区文件列表已经以后端 metadata 为权威来源，删除后全量刷新可以避免前端本地拼装删除后的列表差异。
- 现有状态收敛逻辑已经覆盖“文件从后端列表消失”的场景，复用该逻辑能降低变更面。

Alternatives considered:
- 仅在前端本地乐观移除文件：需要额外维护本地与后端一致性，失败回滚也更复杂。
- 删除后不刷新，仅关闭当前 tab：会留下 Sidebar 列表与真实后端状态不一致的问题。

## Risks / Trade-offs

- [删除运行中依赖文件会导致后续步骤失败] → 通过确认框明确风险，并保持运行时错误直出，不增加前端或后端 fallback。
- [删除流程同时涉及磁盘文件、内存索引和 `file-map.json`，存在部分失败导致短暂不一致的风险] → 将删除逻辑收敛到 `fileStore` 单点实现，并让异常直接暴露给调用方与测试。
- [当前 `output` 同时承载手工新建文件和运行时产物，用户可能误删仍需后续使用的文件] → 确认框统一以“工作区文件”描述，并强调可能影响当前会话或后续执行。
- [Sidebar 文件行从单按钮改为复合结构后，可能破坏现有 hover/selected/active 样式] → 保持原主点击区视觉权重不变，仅为删除按钮增加危险态样式，并补充组件测试覆盖事件边界。

## Migration Plan

1. 先补后端删除 contract 与测试，确保工作区 metadata 能在删除后反映最新状态。
2. 再补前端 API 和 store 删除动作，复用现有工作区状态收敛逻辑。
3. 最后调整 Sidebar 行级结构和样式，并补充组件交互测试。
4. 回滚时可直接移除前端删除入口并回退后端 `DELETE` 路由；已有工作区文件存储结构无需迁移。

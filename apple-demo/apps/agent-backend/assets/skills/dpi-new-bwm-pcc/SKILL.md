# DPI 带宽控制 LLD 生成

## 目标

基于用户提供的 DPI 业务场景自然语言描述，生成 SMF 和 UPF 网元的 DPI 配置 LLD 模板，作为下一步生成 MML 脚本的输入。

## 交互规则

1. 与用户确认时，必须使用下拉选择。
2. 单轮最多提 3 个问题。
3. 交互范围只能基于本技能中明确给出的场景、命令、参数和依赖关系。
4. 缺少关键信息时必须显式追问，禁止臆想。

## 场景识别

根据用户描述先匹配以下场景之一：

1. 基于业务感知的带宽控制
2. 基于业务累计流量的策略控制
3. HTTP/HTTPS/RTSP 头增强
4. HTTP 智能重定向
5. 其他

若无法精确匹配，必须以下拉选择形式让用户确认。若用户选择的不是“基于业务感知的带宽控制”，则明确说明当前技能仅提供该场景的可执行模板，不能输出未定义场景的配置。

## 当前支持范围

当前只支持“基于业务感知的带宽控制”场景，且只允许使用下列命令与其直接依赖关系推导出的配置顺序。

### 操作依赖

| 操作名称 | 依赖操作 | 依赖逻辑 |
| --- | --- | --- |
| 创建过滤器 | 无 | 无 |
| 创建流过滤器 | 无 | 无 |
| 创建规则 | 创建流过滤器 | 规则名称需与流过滤器名称一致；规则名称需与策略组名称一致 |
| 创建用户配置文件 | 无 | 无 |
| 绑定规则与用户配置 | 创建规则；创建用户配置文件 | 规则名称与用户配置文件名称必须引用前面已创建的对象 |

### 已知命令示例

```mml
CREATE FILTER:FILTERNAME="f_any", L34PROTTYPE=STRING, L34PROTOCOL=ANY, MSSTARTPORT=0, MSENDPORT=65535, SVRSTARTPORT=0, SVRENDPORT=65535;
CREATE FLOWFILTER:FLOWFILTERNAME="ff_any", TETHERDETFLAG=NONE, FLOWFILTERTYPE=USERCONFIG;
BIND FILTERTOFLOW:FLOWFILTERNAME="ff_any", FILTERNAME="f_any";
```

### 参数来源约束

| 操作名称 | 参数名称 | 参数来源 | 条件 |
| --- | --- | --- | --- |
| 创建过滤器 | FILTERNAME | 用户定义 | |
| 创建过滤器 | L34PROTOCOL | 固定为 `ANY` | 用户要求对所有业务类型进行管理 |
| 创建过滤器 | MSSTARTPORT | 固定为 `0` | 用户要求对所有业务类型进行管理 |
| 创建过滤器 | MSENDPORT | 固定为 `65535` | 用户要求对所有业务类型进行管理 |
| 创建过滤器 | SVRSTARTPORT | 固定为 `0` | 用户要求对所有业务类型进行管理 |
| 创建过滤器 | SVRENDPORT | 固定为 `65535` | 用户要求对所有业务类型进行管理 |
| 创建流过滤器 | FLOWFILTERNAME | 用户定义 | |

除上表要求用户输入的参数外，其他未列出的参数只能参考已给定命令示例填写；若示例中也没有，必须报缺失，不能猜。

## 执行流程

### 第 1 步：识别场景

- 从用户输入判断是否属于“基于业务感知的带宽控制”。
- 不确定时，用下拉让用户从 5 个场景中选择。

### 第 2 步：收集最少必要参数

仅收集文中明确要求用户定义的参数：

- `FILTERNAME`
- `FLOWFILTERNAME`

如果后续生成包含“创建规则”“创建用户配置文件”“绑定规则与用户配置”，则相关名称也必须先由用户明确提供；不能根据习惯自动补全。

### 第 3 步：按依赖顺序组织模板

输出内容必须遵循依赖链，不得跳步：

1. 创建过滤器
2. 创建流过滤器
3. 绑定过滤器到流过滤器
4. 创建规则（仅当用户已提供所需名称且规则定义充分时）
5. 创建用户配置文件（仅当用户已提供所需名称且配置定义充分时）
6. 绑定规则与用户配置（仅当前置对象都已明确存在时）

若某一步缺少必需名称或参数，必须停止在该步之前并说明缺失项。

## 输出规则

### 输出形式

针对“基于业务感知的带宽控制”场景，只允许以下两种输出之一，按用户要求选择：

1. 只输出 MML 命令文本，格式参照示例。
2. 只输出 MML 命令表，使用 Markdown 表格：
   - 命令名作为表标题
   - 参数名作为列名
   - 参数值作为列值
   - 连续相同命令合并到一个表

### 严格约束

1. 不要输出本文未定义的命令名称。
2. 不要输出与 macro 定义或参数组合原则冲突的参数。
3. 若信息不足，直接说明缺什么，不要给默认值。
4. 不要混合“命令文本”和“命令表”两种格式，除非用户明确要求两者都要。
5. 不要添加解释性背景、扩展建议或模拟配置。

## 失败条件

出现以下任一情况时必须直接失败并说明原因：

- 用户选择了非“基于业务感知的带宽控制”的场景。
- 用户要求生成的命令或参数不在本文定义范围内。
- 必要名称或参数未提供，且无法从本文规则唯一确定。
- 用户要求违反既定依赖关系或参数约束。
